# SoC 策略分析報告 **分析期間：2026-04-26 ~ 2026-04-28** --- ## 任務一：結構性機會偵測 --- ### 機會 1：邊緣裝置「Agentic AI 推論加速」的 SoC 架構機會 #### Step 1：事實觀察 - [SemiEngineering, 2026-04-27]「邊緣 AI 無所不在、代理型、多模態且硬體加速」→ 觀察到的事實是：產業論述已從「邊緣 AI 推論」轉向「邊緣 Agentic 推論」，且強調必須硬體加速。[來源] - [Android Authority, 2026-04-27]「Google Gemma 4 在您的口袋中：如何完全離線執行最新 AI」→ 觀察到的事實是：基礎模型廠商開始主動推出可在手機端完全離線執行的版本。[來源] - [科技新報, 2026-04-27]「Meta 員工自建 AI 用量排行榜秀 30 天就破 60 兆 Token，但資料外流迅速撤下」→ 觀察到的事實是：企業內部 LLM 使用量爆發後，立刻引發資料外流事件並被撤下。[來源] - [TechCrunch AI, 2026-04-27]「OpenAI 可能在製造手機，用 AI agents 替代應用程式」→ 觀察到的事實是：應用層玩家開始往硬體層移動，視 Agent 為新 UI 主軸。[來源] - [科技新報, 2026-04-27]「打破軟體業獲利定律！AI 越多人用成本越高」→ 觀察到的事實是：雲端推論的單位成本不會隨規模下降，反而隨使用增加而上升。[來源] #### Step 2：條件變化分析 - 哪個條件改變？2025 下半年至 2026 Q1 期間，**「小型化高品質模型」（Gemma 4、Phi、Llama 3.2 等 3B-8B 級）的能力，跨過了「足以執行多步驟 Agent 任務」的門檻**。在 2024 年初，端側模型主要只能做翻譯/摘要等單步任務。 - 性質：**技術性 + 經濟性雙重變化** - 技術：模型壓縮 + 量化使 8B 模型可在 12GB RAM 手機跑通 - 經濟：雲端 Token 推論的邊際成本不下降（科技新報，2026-04-27），使「將 Agent 工作負載推到端側」從「不重要的選項」變成「必要的成本管控手段」 - 多久前不存在？**18 個月前（2024 Q3）**：當時端側模型的工具調用準確率 < 60%，無法可靠完成 Agent 任務；雲端推論成本曲線當時仍預期會隨規模下降。 - [推斷]（基於來源事實的合成） #### Step 3：趨勢導出 - Q2 機制：條件變化透過兩條路徑改變產業： 1. **經濟路徑**：雲端 Agent 成本不規模化 → 應用商必須將部分推論下沉到端側 → 端側 SoC 成為 Agent 經濟的成本控制點 2. **隱私路徑**：Meta 60 兆 Token 事件顯示「資料外流」是 Agent 規模化的核心障礙 → 企業/政府客戶要求 Agent 在裝置內完成 → 端側推論不是選項而是合規要求 - Q3 為何此時足夠？2024 年端側模型品質不足，即使有經濟誘因也無法部署；2026 Q1 模型品質已過門檻，加上資料外流事件（Meta）創造了具體的合規壓力臨界點，三條件同時成熟。 - Q4 反事實：如果端側模型品質仍停在 2024 水準，即使雲端成本曲線不下降，企業也只能容忍——因為沒有可行替代。趨勢不會發生。 - **趨勢：Agent 工作負載的「端雲分配比例」，正從 2024 年的「>95% 在雲」結構性轉向 2028 年的「30-50% 在端」** 驗證： - ✅ 宏觀性：影響整個運算產業 - ✅ 長期性：5 年以上（雲端成本曲線不會在 5 年內反轉） - ✅ 不可逆性：合規壓力一旦形成不會消失 #### Step 4：市場新機會浮現 - Q1 競爭條件改變：原本端側 NPU 的設計目標是「執行單一神經網路推論」（影像分類、降噪），現在目標變成「執行多步驟、多工具調用、長 context 的 Agent 推論流」。這是兩個不同的工作負載特徵： - 舊：固定 batch、固定形狀、低延遲單次 - 新：變動 context（4K-128K token）、KV cache 管理、頻繁 tool call 中斷 - Q2 機制：現有 SoC 的 NPU 架構（高通 Hexagon、聯發科 APU）是為前者優化，**KV cache 管理與動態形狀支援不足**，導致跑 Agent 時實際吞吐量遠低於理論 TOPS 值。新機會是「為 Agent 工作負載重新設計記憶體階層的端側 SoC IP」。 - 為何現有競爭者沒解決？ - **架構鎖定**：高通/聯發科的 NPU 架構過去 3 代都是固定形狀優化，重新設計記憶體子系統需 18-24 個月 - **利益衝突**：他們的旗艦 SoC 路線圖已宣告，中期無法回頭 - Q4 反事實：如果 Agent 不下沉到端側，現有 NPU 架構就夠用，這個機會不存在。 - **市場新機會：為「Agent 推論工作負載」優化的端側 SoC IP / 子系統（針對 KV cache、動態 context、tool-call 中斷恢復）**，目標是手機/PC/車用 SoC 的 NPU 子系統授權或整合。 #### Step 5：目標客群與 JTBD 問題 A：為什麼是「二線手機 SoC 廠商」（紫光展銳、Google Tensor 團隊、三星 Exynos），而不是其他？ - ❌ 排除高通：他們是現有架構的鎖定者，採用新 IP 等於承認自己錯了 - ❌ 排除聯發科：類似理由 + 他們有自研 APU 路線 - ❌ 排除 Apple：垂直整合，不外購 IP - ✅ **二線手機 SoC 廠商**：他們在旗艦 NPU 性能上已落後高通/聯發科 1-2 代，需要差異化突破點才能說服 OEM 採用 問題 B：結構性條件——他們**正在被高通/聯發科從 OEM 訂單中擠出**，需要「不在 TOPS 競賽中對拚，而在 Agent 場景中勝出」的差異化敘事。Google Tensor 尤其有此壓力（自家 Gemma 跑不順是公關災難）。 問題 C：有能力解決的廠商為何沒做？ - 高通/聯發科：**選擇不做**（利益衝突）——若公開承認需重新設計記憶體子系統，等於宣告現有旗艦的 Agent 性能不足，會傷害現有定價 - ARM/SiFive：**商業模式不符**——他們提供 CPU/GPU IP，NPU 子系統不是核心業務 - **JTBD：二線手機 SoC 廠商在規劃 2027-2028 旗艦 SoC 時，需要在「總晶片面積不增加、功耗不增加」的約束下，將 7B 級 Agent 的端到端延遲降低 3 倍以上，作為 OEM 採購的差異化說服點** #### Step 6：痛點 - 具體環節：**KV cache 在端側 DRAM 與 NPU SRAM 之間的搬移**——當 context 從 4K 增長到 32K 時，現有 NPU 的記憶體頻寬瓶頸使每 token 延遲呈非線性增長 - 為何現有方案無法改善？ - 不是「未優化」，而是「NPU 與 DRAM 的物理介面（LPDDR5X 通道數）在現有 SoC 是固定的」 - 重新設計需要修改記憶體控制器與 NPU 之間的 fabric，影響整顆 SoC 的 floorplan - Q4 反事實：如果現有 SoC 能解決這個阻礙（例如 LPDDR6 普及），客群不會找新方案。但 LPDDR6 至少 2027 才量產 → 痛點窗口存在 2 年。 - 痛點深度：高。二線廠商失去一個 OEM 旗艦 socket 的損失約 USD 50-200M 訂單。 - **核心痛點：在現有 LPDDR 頻寬下，NPU 無法支撐長 context Agent 的可接受延遲** #### Step 7：客戶價值 解決方案：**KV cache 感知的 NPU 記憶體子系統 IP**——分層快取、推測性 prefetch、KV 量化感知的搬移排程。 - 機制：透過將 KV cache 訪問模式從「隨機」轉為「可預測」，將有效記憶體頻寬利用率從約 40% 提升至約 75%（[推斷]，需基準測試驗證） - 效益：32K context 下，token 生成延遲降低 2-3 倍；總晶片面積增加 < 3% - 成本：IP 授權費 + 整合工程 6-9 個月 - 效益 > 成本機制：與失去一個 OEM socket 的機會成本（USD 50-200M）相比，IP 授權費（USD 5-20M）顯著划算 - 競爭壁壘：高通/聯發科若要追，需 18-24 個月架構修改週期 → 窗口約 24-30 個月 - Q4 反事實：若 Agent 不下沉到端側（痛點不存在），此 IP 無價值 #### Step 8：商業價值 商業模式：**NPU 子系統 IP 授權（per-chip royalty）+ 整合服務費** - 擷取機制：每顆採用此 IP 的 SoC 收取 USD 0.5-1.5 royalty；前期整合服務費 USD 3-5M / 客戶 - 定價權來源：架構壁壘（KV cache 子系統設計知識）+ 高轉換成本（一旦整合進 SoC，下一代不會輕易換） - 轉換成本組成：6-9 個月整合工程、矽前驗證、後續驅動軟體相容性 - 護城河動態： - 加深機制：每多一個客戶 → 更多真實 Agent workload 的調優資料 → IP 越優化 - 削弱威脅：LPDDR6 量產（2027）、高通新架構（2028）會降低痛點強度 - 護城河頂峰：**2027 H2** - 商業價值上限：若 24 個月內取得 3-5 家二線 SoC 廠商客戶，每家年出貨 30-80M 顆 → 累計 royalty USD 150-400M - 市場規模：[未驗證，建議搜尋確認 2027 全球非高通/聯發科手機 SoC 出貨量] --- ### ✅ 任務一因果鏈完整性審核（機會 1）

[The New Stack, 2026-04-28]「Red Hat 的 OpenClaw 維護者大幅提升企業 Claw 部署的安全性」「Tank OS 將 OpenClaw AI agents 放入容器中」→ 軟體層已試圖用容器隔離 Agent。[來源] - [科技新報, 2026-04-27]「Meta 員工 60 兆 Token 排行榜資料外流迅速撤下」→ 即使內部工具，Agent 處理的資料外流速度遠超人類監控。[來源] - [The New Stack, 2026-04-27]「透過 Claude Managed Agents，Anthropic 希望為您運營 AI Agent」→ 業界正將 Agent 運營外包，但同時擴大攻擊面。[來源] #### Step 2：條件變化分析 - 條件改變：**Agent 從「建議者」變成「執行者」**——2024 年 Agent 主要產出文字建議由人類執行；2026 年 Agent 直接執行 API 呼叫、檔案操作、資料庫變更 - 性質：**技術性 + 監管性**（Manitoba 禁青少年用 AI 是早期監管信號） - 多久前不存在？**12 個月前**，主流 Agent 框架（LangChain 等）還主要是 RAG，少有實際 write 權限 - [來源 + 推斷] #### Step 3：趨勢導出 - Q2 機制：Agent 直接執行 → 軟體 sandbox（容器、VM）保護不足，因為 Agent 持有合法憑證並做合法 API 呼叫，行為層面難區分惡意/誤操作 → 必須在硬體層做能力限制（capability-based security） - Q3 為何此時足夠？9 秒刪庫事件 + Meta 外流事件 + 監管啟動三點同時發生，企業法務開始要求「可審計的硬體級隔離」 - Q4 反事實：若 Agent 仍只是建議者（人類最終執行），人類可以是最後防線，硬體隔離無必要 - **趨勢：企業基礎設施將從「軟體定義安全」結構性轉向「硬體強制可審計的 Agent 執行環境」** 驗證：✅ ✅ ✅ #### Step 4：市場新機會 - Q1 改變：CPU/SoC 的 TEE（TrustZone、SGX、TDX）原本為保護「資料機密性」設計，現需轉為保護「執行能力的最小化原則」（Agent 只能做被授權的事） - Q2 機制：現有 TEE 模型是「保護內部不被外部偷看」，Agent 場景需要「保護外部不被內部誤動」——這是反向的威脅模型，需要新的硬體 primitive - 現有競爭者為何沒解決？ - Intel/AMD：**架構鎖定** + 企業伺服器市場為主，端側 Agent 場景不是優先 - ARM CCA：方向接近但仍針對機密運算，Agent 能力限制不是設計目標 - Q4 反事實：若 Agent 不執行高風險操作，TEE 無需擴展 - **市場新機會：針對 Agent 執行的「能力代幣化」（capability token）硬體 IP——每個工具呼叫須持有硬體簽發的 capability，過期或越權自動拒絕** #### Step 5：目標客群與 JTBD 問題 A：為什麼是「企業端 PC / Workstation SoC 客戶」（Dell/HP/Lenovo 商用線採用的 SoC），而不是其他？ - ❌ 排除手機：個人 Agent 風險規模小，企業不買單 - ❌ 排除雲端伺服器：超大規模雲商自研（AWS Nitro 思維） - ❌ 排除消費 PC：消費者支付意願低 - ✅ **企業 Workstation SoC**：企業已被刪庫事件嚇到，且合規預算充足 問題 B：結構性條件——**企業合規部門在 9 秒刪庫事件後啟動 Agent 部署 review**，採購規格將寫入「硬體級 Agent 隔離」要求。 問題 C：為何沒人做？ - Intel/AMD：**選擇延後**——伺服器 TDX 還沒推完，PC 端優先級低 - 雲商自研：**商業模式不符**——Nitro 思路是雲側，不會出貨給企業 PC - **JTBD：企業 IT 在 2027 採購 Workstation 時，需要在 RFP 中能勾選「硬體級 Agent capability 限制」項目，以滿足內部風險委員會要求** #### Step 6：痛點 - 環節：**Agent 在執行 tool call 時，作業系統無法在不增加 100ms+ 延遲下強制驗證該呼叫是否在授權範圍內** - 為何無法改善？軟體層驗證需穿越 kernel + 安全模組，每次 tool call 開銷大；硬體層目前無 capability 驗證 primitive - Q4 反事實：若延遲不重要，軟體驗證夠用——但 Agent UX 對延遲敏感 - **核心痛點：可審計的能力限制與 Agent 互動延遲之間的零和取捨** #### Step 7：客戶價值 解決方案：**SoC 內建 Capability Verification Unit（CVU）**——硬體驗證 capability token，延遲 < 5μs - 機制：將 capability 檢查從 kernel syscall 路徑移至硬體 fast path - 效益：Agent tool call 延遲增加 < 1%；同時產生硬體簽署的審計日誌（合規可用） - 成本：IP 授權 + SoC 整合 - 競爭壁壘：Intel/AMD 跟進需 24-36 個月（PC 平台週期長） - Q4 反事實：若刪庫類事件不發生，企業合規不會強制此項 #### Step 8：商業價值 商業模式：**IP 授權 + 合規認證生態（類似 TPM 的長尾收益）** - 定價權：硬體 + 規範雙重壁壘 - 護城河動態： - 加深：合規規範一旦寫入（如 NIST），形成事實標準 - 削弱：Intel/AMD 跟進 - 頂峰：**2028（合規規範定型前的 standard-setting 窗口）** - 商業價值上限：若進入企業 PC SoC 主流路線圖，潛在 royalty USD 100-300M / 年 - 市場規模：[未驗證] --- ### ✅ 任務一因果鏈完整性審核（機會 2）

# 任務一因果鏈反事實審核 ## 機會 1：邊緣裝置 Agentic AI 推論加速 SoC
#箭頭反事實：拿掉 A 後 B 是否消失？消失的機制判定
1.1模型小型化過門檻 → 端側 Agent 可行消失，因為若 3-8B 模型仍停在 2024 工具調用 < 60% 準確率，端側無法可靠完成多步驟任務，無論硬體多強都不可行模型品質是 Agent 可行性的必要條件✅
1.2雲端推論成本不規模化 → 工作負載下沉到端側消失，因為若雲端邊際成本隨規模下降（傳統軟體經濟），應用商無動機承擔端側部署的工程成本經濟誘因消失則路徑選擇不啟動✅
1.3Meta 60 兆 Token 外流事件 → 企業合規要求端側執行**不消失**——資料外流的合規壓力來源是「LLM 處理敏感資料」這個結構性事實，即使沒有 Meta 這起特定事件，GDPR、SOC2、金融業內規本身就會驅動同樣要求Meta 事件只是「觸發報導關注」，不是「創造合規壓力」❌
1.4工作負載下沉 → NPU 架構需重設計消失，因為若工作負載仍是固定形狀單次推論，現有 NPU 架構（Hexagon、APU）足夠，無重設計動機Workload 特徵改變才是架構壓力來源✅
1.5高通/聯發科旗艦路線圖鎖定 → 二線廠商有差異化窗口消失，因為若大廠能即時調整架構，二線廠商無法以「Agent 延遲」做差異化敘事——大廠下一代就追上架構鎖定創造的時間差是窗口的物理基礎✅
1.6二線廠商面臨 OEM 訂單流失壓力 → 採用此 IP 動機消失，因為若二線廠商市佔穩定無壓力，他們會選擇低風險的漸進改良而非賭新 IP生存壓力是採用新架構風險的前提✅
1.7OEM 採購標準轉向 Agent 延遲 → IP 商業價值成立**部分不消失**——即使 OEM 採購標準不轉向，只要終端使用者能感知 Agent 延遲差異（類似當年攝影差異），市場壓力仍會傳導採購標準不是唯一傳導路徑，使用者口碑也是⚠️
