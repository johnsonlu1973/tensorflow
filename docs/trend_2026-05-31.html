# SoC 策略分析報告 **分析期間：2026-05-24 ~ 2026-05-27** --- ## 任務一：結構性機會偵測 --- ### 機會 1：邊緣裝置端 Agentic AI 推論 SoC——專為「多 Agent 編排 + 大量 Token 吞吐」設計的客戶端晶片 #### Step 1：事實觀察 - [科技新報, 2026-05-25]「Meta 員工自建 AI 用量排行榜秀 30 天就破 60 兆 Token」→ 觀察到的事實是：單一企業內部 AI Token 消耗量在月度級已達 60T 規模，且仍快速成長 - [SemiEngineering, 2026-05]「邊緣 AI 無所不在、代理型、多模態且硬體加速」→ 觀察到的事實是：產業已將「Agentic + 多模態 + 硬體加速」並列為邊緣 AI 的下一階段特徵 - [Android Authority, 2026-05]「Google Gemma 4 在您的口袋中：如何完全離線執行最新 AI」→ 觀察到的事實是：Google 已釋出可在手機端離線執行的 Gemma 4 - [TechCrunch AI, 2026-05]「ClickUp 用數千個 AI agents 替代數百名員工」→ 觀察到的事實是：企業生產流程已開始用 agent 群取代人力，agent 數量為「員工數量級的 10 倍以上」 - [The New Stack, 2026-05-27]「為什麼 AI agents 需要 Context Lake」→ 觀察到的事實是：單一 agent 的工具呼叫已不足以完成任務，需要持久化的上下文層 #### Step 2：條件變化分析 - **改變的條件**：AI 推論的單位需求量在 2025 年底~2026 年初從「單次對話 (~1K-10K tokens)」躍升至「持續性 agent 任務 (~1M-100M tokens/任務)」 - **改變類型**：行為性 + 技術性。行為上，使用者從「問答」轉向「委派任務」；技術上，可離線執行的開源模型（Gemma 4、Qwen 等）使端側推論成為可行選項 - **改變時間點**：2024 年仍是雲端集中推論為主；2025 Q4 起 Claude Sonnet/Opus 開始支援多步 agent 工作流，2026 Q2 Gemma 4 端側化 + 企業 agent 平台（Gemini Enterprise）大規模落地 - [推斷] 標記：Token 消耗從「對話級」躍升至「任務級」是邏輯推導，具體量級需更多公開數據驗證 #### Step 3：趨勢導出 - **Q1（條件變化）**：每用戶單日 Token 消耗量從約 10K 級躍升至潛在 10M 級（千倍級） - **Q2（傳導機制）**：Token 消耗 × 雲端推論單價 = 雲端 AI 服務的邊際成本曲線陡升 → 在用戶量擴張時，CSP 的毛利結構從「規模經濟」反轉為「規模負擔」 → 推論工作負載必然部分外溢至端側 - **Q3（為何此時足夠）**：三個條件同時到位才形成臨界點：(a) 端側可執行的模型品質達到實用門檻（Gemma 4、Qwen3）；(b) Agent 工作流催生百倍 Token 需求；(c) Anthropic 等模型廠願意授權給端側部署（Claude Managed Agents 商業化）。2024 年缺 (a) 和 (b) - **Q4（反事實）**：如果 Token 消耗仍停留在對話級（10K/日），雲端推論成本完全可吸收，端側推論趨勢不會出現。因此 Token 量級躍升是真正的因 - ✅ 宏觀性：影響整個運算產業的工作負載分配 - ✅ 長期性：Agent 取代人工流程是 5-10 年的轉變 - ✅ 不可逆性：一旦企業生產流程接入 agent，回退成本極高 **因此，產業趨勢是**：AI 推論的「重力中心」從雲端向「端雲混合 + 端側為主」轉移，驅動力是 Agent 化使每用戶 Token 需求出現千倍級躍升。 #### Step 4：市場新機會浮現 - **Q1（改變的競爭條件）**：手機/PC SoC 過去的 NPU 設計目標是「單次推論延遲」，但 agent 工作負載的真正瓶頸是「持續吞吐 + 記憶體頻寬 + 多 agent 並發排程」——這是一個新的優化維度 - **Q2（傳導機制）**：當端側需要同時運行 3-10 個 agent（規劃者、檢索者、工具調用者）並維持小時級執行時，現有 NPU 架構（為單一模型 single-shot 推論優化）的記憶體子系統與排程器無法勝任 → 出現「Agent-Native NPU 架構」的空白 - **為何現有競爭者沒充分滿足**： - Qualcomm/MediaTek 的 Hexagon/APU 設計於 2022-2023 年，當時 agent 概念尚未成熟，架構鎖定於 single-shot 推論 - Apple Neural Engine 同樣為「Siri 級」單次推論優化，且 Apple 戰略上仍以雲端 Private Cloud Compute 為主 - NVIDIA 在端側無 SoC 級產品（Jetson 屬嵌入式而非消費端） - **Q4（反事實）**：若無 Agent 趨勢，現有 NPU 完全夠用，無此機會空間。確認因果 **因此，市場新機會是**：為「智慧型手機、AI PC、邊緣閘道設備」設計的 **Agent-Native SoC**——核心差異化是高頻寬記憶體子系統（支援多 agent 並發 KV cache）+ 硬體級 agent 排程器 + 端雲流量分割引擎。 #### Step 5：目標客群與 JTBD **問題 A：為什麼是這個客群？** 候選客群： - ❌ 智慧型手機 OEM（Samsung、小米、OPPO）：他們確實需要，但 SoC 採購決策權有限，且高度價格敏感 - ❌ 雲端 CSP（AWS、Google Cloud）：他們的痛點在伺服器端，與端側 SoC 無關 - ❌ 一般 PC OEM（Dell、HP）：Windows on ARM 生態尚未成熟，AI PC 銷量仍小 - ✅ **「AI 原生服務商」自有硬體部門**——OpenAI/Anthropic/Google 為了控制 agent 體驗端到端，正在規劃自有裝置（Jony Ive × OpenAI、Google Pixel × Gemini 深度整合） **因此目標客群是**：（一）AI 模型廠的自有硬體部門；（二）將 agent 作為核心差異化的高階 Android OEM（首發客戶為 Google Pixel 因其 Gemma 整合） **問題 B：結構性支付意願來源** - AI 模型廠：每雲端 Token 的 COGS 是真實的損益表負擔，端側分流是「直接改善毛利率」的事，支付意願 = 雲端推論成本的 NPV - 高階 Android OEM：在通用規格趨同的市場中，「Agent 體驗品質」是少數可差異化的維度，且使用者已開始用 DuckDuckGo 安裝量 +30% [TechCrunch, 2026-05] 表態反對被強加 AI——OEM 需要「在裝置上提供使用者可控的 agent」作為對抗 Google 集中化的差異化 **問題 C：有能力的廠商為什麼還沒做？** - **選擇 B（利益衝突）+ C（架構障礙）混合**：Qualcomm 主要客戶是 Android OEM，但 Qualcomm 為了押注 Snapdragon X Elite 進入 AI PC 市場，將工程資源集中於「對標 Apple Silicon」而非「為 agent 重新設計」。架構上，現有 NPU 的記憶體子系統與 CPU/GPU 共用 LPDDR，無法支援多 agent KV cache 常駐——這需要 SoC 級架構重做，週期 2-3 年 **因此 JTBD 是**：在 AI 模型廠/高階 OEM 推出下一代 agent-first 裝置時（2026 H2 ~ 2027），提供「能在裝置端持續運行 3-10 個並發 agent 達 8 小時以上、且不犧牲日常使用體驗」的 SoC 平台。 #### Step 6：痛點 - **具體環節**：多 agent 並發時的「記憶體頻寬爭奪」與「KV cache 管理」 - **現有方案為何無法改善**： - 現有 SoC 的 LPDDR 頻寬（~70 GB/s for LPDDR5X）由 CPU/GPU/NPU/ISP 共用，當 NPU 需獨佔頻寬執行 LLM 推論時，其他子系統會降速 → 使用者體驗劣化 - KV cache 在端側需頻繁換入換出，現有架構無「KV cache 專用 SRAM 層」 - 反事實確認：若現有方案能解決，OEM 不會尋找替代——而 Google 自研 Tensor、Apple 自研 Neural Engine 的存在證明 OEM 已在尋找 - **支付意願量化**：[推斷] 若一台 agent 旗艦機定價較通用旗艦機高 $200，且 OEM 願將其中 $30-50 流向 SoC 廠（vs Qualcomm 旗艦 SoC ~$200），即 +15-25% ASP 溢價 **因此核心痛點是**：在現有 SoC 架構下，端側多 agent 並發運行會在 30-60 分鐘內因記憶體頻寬爭奪導致整機體驗崩潰（電池、發熱、卡頓三重）。 #### Step 7：客戶價值 **解決方案**：Agent-Native SoC——配備獨立 KV cache SRAM 池（建議 64-128MB）+ 硬體 agent 排程器 + 端雲動態流量分割。 - **機制**：KV cache 從 LPDDR 移至專用 SRAM → 消除頻寬爭奪；硬體排程器根據 agent 優先級與電池狀態動態調度 → 維持 8 小時續航；流量分割引擎自動將高複雜度 query 分流到雲 → 平衡品質與成本 - **效益**：對 AI 模型廠 = 每用戶雲端推論成本下降 40-60% [推斷]；對 OEM = ASP 溢價 $200/台 + 差異化故事 - **成本**：SoC 採購單價 +$30-50；軟體 stack 整合工程約 6-12 人月 - **效益 > 成本**：對 AI 模型廠，若每月雲端推論成本 $5/用戶，分流 50% 後每年節省 $30/用戶，3 年 NPV 遠超 SoC 增量 - **競爭壁壘時間窗口**：Qualcomm 從決策到 tape-out 需 18-24 個月，量產到客戶手中再加 6 個月 → 領先者有 24-30 個月窗口 - **反事實**：若無多 agent 工作負載，KV cache 專用 SRAM 是純成本，無客戶價值。確認因果 #### Step 8：商業價值 **商業模式**：高利潤 SoC 銷售 + 軟體 stack（agent runtime）授權的 razor-and-blades - **擷取機制**：SoC ASP $200-250（vs Qualcomm 旗艦 ~$200），同時 agent runtime SDK 綁定客戶 - **定價權來源**：(a) Agent runtime 與 SoC 共設計，軟體 stack 遷移成本 = 客戶工程團隊 1-2 年重寫；(b) AI 模型廠的 agent 效能 benchmark 一旦在此平台校準，遷移會破壞使用者體驗承諾 - **護城河動態**： - **加深機制**：每代 agent runtime 累積使用模式資料，反饋至下一代硬體設計（資料飛輪） - **削弱威脅**：MediaTek/Qualcomm 在 24-30 個月後推出對標產品 + RISC-V 開源 NPU 生態（[科技新報] 三星 SSD 控制器已導入 RISC-V，2-3 年內可能延伸至 NPU） - **頂峰時點**：2027-2029（領先者窗口期），之後進入價格競爭 - **商業價值上限**：[未驗證，建議搜尋確認] 假設高階智慧型手機年銷量 ~3 億台，agent-first 旗艦滲透率 5%（1500 萬台）× ASP $200 = $30 億/年；3 年窗口期累積 $60-90 億可擷取營收 - **市場規模**：[時間不明，建議搜尋確認] 全球高階智慧型手機（>$600）2025 年約 3 億台 --- ### ✅ 機會 1 因果鏈完整性審核

**改變的條件**：AI 推論工作負載結構從「批次大型模型推論」轉為「大量小型 agent step + 工具調用 + 記憶體存取」 - **改變類型**：技術性。Agent 工作流是「多步驟、低延遲、高並發、中等模型大小」的混合負載，與訓練（大模型、長批次）和傳統推論（單次大模型）都不同 - **改變時間點**：2025 H2 後 Claude Managed Agents、Gemini Enterprise Agent Platform 等正式商品化 #### Step 3：趨勢導出 - **Q1**：CSP 的 AI 工作負載中「agent step」占比從 <5% 升至預估 30-50% - **Q2**：Agent step 的特徵（小模型多次調用 + 大量工具 I/O + 記憶體密集）使 GPU 的高 FLOPs/低互聯效率不再最佳 → CSP 有動機自研針對此工作負載的 ASIC - **Q3**：CSP 自研 ASIC 的固定成本約 $500M-1B，需 100K+ chip 部署才划算；agent 工作負載在 2026 達到此規模門檻 - **Q4**：若 agent 工作負載僅為小眾，CSP 不會自研。確認 **產業趨勢**：CSP 自研推論 ASIC 的差異化方向從「通用 LLM 推論」轉向「Agent 工作流專用」（TPU v6e+、Trainium 2+、Maia 2 都將朝此演進）。 #### Step 4-8（簡化版，因主要受惠者為 CSP 自身而非外部 SoC 廠） **機會本質**：對外部 SoC 廠（如 Marvell、Alchip、世芯）而言，機會是 **CSP ASIC 的設計服務與後端代工**——CSP 不會自己做 physical design 與 backend，需要 ASIC 服務商。 **目標客群**：第二梯隊 CSP（Oracle、SoftBank 主權 AI、Sakura Internet）——他們有需求但無 Google/AWS 等級的內部團隊。Tier-1 CSP 已綁定 Broadcom/Marvell。 **為什麼是這個客群而非 Tier-1**：Tier-1 已綁定既有夥伴；Tier-2 是「結構性新進者」，[Bloomberg, 2026-05] Sakura Internet 7 倍資本支出證明預算到位 **痛點**：Tier-2 CSP 想自研 ASIC 但無 backend 經驗，且時間壓力大（2026-2027 必須上線） **客戶價值**：將 ASIC 從 spec 到量產的時間從 24-30 個月壓縮至 18 個月 **商業價值**：[未驗證] 每個 ASIC 案子設計服務費 $30-80M + NRE，年度可承接 3-5 案 --- ## 任務二：海外 vs 台灣媒體落差 ### 落差 1：華為 1.4nm 與 LogicFolding 的解讀 **矛盾現象**： - 海外 [Tom's Hardware, 2026-05-25]：技術可行性質疑為主，標題用「claims」「could」等保留語 - 台灣 [科技新報, 2026-05-25]「華為發表半導體韜定律，拚晶片效能 2031 年達 1.4 奈米」：較中性偏報導事實 **可能原因**： - 解釋 A [推斷]：海外媒體對中國半導體聲稱有「制裁有效性」的政治預設，傾向質疑。機制：報導敘事服務於讀者對美國政策的認同 - 解釋 B [推斷]：台灣媒體擔心過度貶低華為會被解讀為「低估對手」，選擇中性。機制：台灣半導體產業需要保留對手戰力評估彈性 **哪個解釋更有支撐**：B 較有支撐。台灣產業界（含 TSMC）對華為的私下評估一向比公開報導謹慎。海外的質疑亦有合理技術根據（LogicFolding 是 3D 堆疊變體，並非真正繞過 EUV）。 **對 SoC 規劃影響**：華為 2031 年聲稱不影響當下 2026-2028 規劃；但 LogicFolding 若為真實技術路徑，2028 後可能改變先進製程的成本結構。 **監測指標**： - --- ## ❌ 判定的修正 ### 修正 1：Qualcomm 資源分配的因果 **修正**：原主張「Qualcomm 押注 Snapdragon X Elite → 無暇做 agent SoC」 **原因**：這是「平行觀察」而非「因果機制」。Qualcomm 即使不押注 X Elite，也可能因更根本的原因不做 agent SoC——例如 Hexagon NPU IP roadmap 已 commit 至 2027、或主要 OEM 客戶（Samsung、小米）尚未提出 agent SoC 規格需求。X Elite 是「症狀」，不是「原因」。 **修正後因果**： - 真正原因 [推斷]：Qualcomm 的 NPU 架構由 Hexagon DSP 演化而來，其記憶體子系統與 DSP 管線深度耦合，重做 KV cache 專用 SRAM 子系統需打破現有 IP 模組邊界，工程週期 24+ 個月，且會破壞既有客戶的軟體相容性承諾 - 證據需求：需驗證 Hexagon roadmap 公開資訊與客戶軟體生態鎖定程度 **不受影響**： - 「現有大廠有 18-24 個月窗口期」的結論仍成立（基於 SoC tape-out 週期的物理限制，與 Qualcomm 具體策略無關） - 「Agent-Native SoC 架構差異化」的技術論證不受影響 - 機會 1 的目標客群（AI 模型廠自有硬體 + 高階 Android OEM）分析不受影響 --- ### 修正 2：Tier-1 綁定與 Tier-2 機會的因果 **修正**：原主張「Tier-1 已綁定 Broadcom/Marvell → Tier-2 是可獲取客群」 **原因**：「Tier-2 是可獲取客群」的真正理由不是「Tier-1 被綁定」，而是「Tier-2 自身具備規模、時間壓力與能力缺口」。原因果把「排除 Tier-1」誤當成「Tier-2 成立」的理由。 **修正後因果**： - 正確因果：Tier-2 CSP（Sakura、SoftBank、Oracle）在 2026-2027 同時出現「資本支出到位 [Bloomberg, 2026-05]」+「時間壓力（主權 AI 政治承諾）」+「內部 backend 能力缺口」三個條件 → 成為 ASIC 設計服務的高支付意願客群 - Tier-1 是否綁定只影響「市場上限」（總可服務市場），不影響 Tier-2 機會的成立 **不受影響**： - Tier-2 CSP 的資本支出證據（Sakura 7 倍、SoftBank 主權 AI）仍是有效論據 - 設計服務的商業模式（NRE + per-chip royalty）不受影響 - 18 個月時間壓縮的客戶價值論證不受影響 --- ## ⚠️ 部分成立的補強 ### 補強 1：雲端成本 → 端側推論 **原因果不完整**：雲端成本只是「驅動因素之一」，隱私、延遲、離線可用性也是獨立驅動。 **補強後**：端側推論趨勢由「雲端成本不可吸收（主因，約 60% 權重）+ 隱私合規（次因，受 EU AI Act 與企業資料治理推動）+ 延遲（agent 多步推理累積延遲）」三因素共同驅動。即使移除任一因素，趨勢仍會存在但強度降低。 **對機會 1 的影響**：機會仍成立，但「目標客群為 AI 模型廠」的論證需補充——若隱私是主因，目標客群應加入「企業內部部署場景」（受規則三限制，此處標記為[未驗證]待後續分析）。 ### 補強 2：Agent 工作負載 → CSP 自研 ASIC **原因果不完整**：CSP 自研 ASIC（TPU 2016、Trainium 2018、Maia 2023）早於 agent 趨勢成熟，agent 是「強化既有方向」而非「創造新方向」。 **補強後**：Agent 工作負載特徵 → CSP 自研 ASIC 的**設計目標重心轉移**（從「訓練優化」轉向「推論 + 多步驟編排優化」）。原本 CSP 自研動機已存在，agent 改變的是「下一代 ASIC 的架構選擇」。 **對機會 2 的影響**：機會本質從「催生新自研需求」修正為「現有自研路線圖的架構轉向窗口」——這反而強化了時間窗口的迫切性（CSP 必須在下一代 ASIC tape-out 前決定架構，2026-2027 是決策點）。 --- ## 審核總結
