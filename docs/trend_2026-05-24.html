# SoC 策略分析報告｜2026 年 5 月第 3 週 --- ## 任務一：結構性機會偵測 --- ### 機會 1：「邊緣 AI Agent 執行硬體」——本地化、低延遲、多模態的 Agent 推理 SoC #### Step 1：事實觀察 - [SemiEngineering，2026.05]「邊緣 AI 無所不在、代理型、多模態且硬體加速」→ 事實是：產業媒體開始將「Agentic」與「Edge」並列為同一個技術趨勢的描述詞。[來源] - [Android Authority，2026.05]「Google Gemma 4 在您的口袋中：如何完全離線執行最新 AI」→ 事實是：Google 推出可離線執行的 Gemma 4 模型，並由媒體以「口袋」描述其執行位置。[來源] - [The New Stack，2026.05.19]「Anthropic 推出 MCP tunnels 和自託管沙箱以強化 AI agent 基礎設施」→ 事實是：Anthropic 在開發者大會推出「自託管沙箱」，意味 Agent 執行環境正從純雲端延伸到客戶端。[來源] - [科技新報，2026.05]「Meta 員工自建 AI 用量排行榜秀 30 天就破 60 兆 Token」→ 事實是：單一公司內部 30 天 token 消耗達 60 兆，顯示集中式雲端推理成本曲線陡峭。[來源] - [TechCrunch AI，2026.05.18]「南韓 LetinAR 正在開發 AI 眼鏡背後的光學技術」+ [MIT Tech Review，2026.05.18]「Anduril 和 Meta 開發軍用智慧眼鏡」→ 事實是：穿戴式 AI 終端（眼鏡）同時在消費、軍用兩條路徑被推進。[來源] #### Step 2：條件變化分析 - **改變的條件**：「在端側執行有用 Agent 所需的模型尺寸」——從 2023 年的 70B+（必須雲端）下降到 2026 年的 4B 級可離線執行（Gemma 4 等）。 - **改變類型**：技術性（模型蒸餾、量化、MoE 架構成熟）＋ 經濟性（雲端 token 成本對企業變成可量化壓力，如 Meta 60 兆 token 事件外溢）。 - **時間點**：此條件在 2024 上半年仍不成立（端側模型品質不足以執行 Agent 多步推理）；2025 下半年達臨界（Gemma 3 / Phi-3 / Llama 3.2 證明可行）；2026 Q2 被生態廠（Google、Anthropic）正式以產品形式對外宣告。 - [推斷]：模型尺寸/品質曲線跨越「Agent 可用門檻」是 12-18 個月內的事件。 #### Step 3：趨勢導出 - **Q1（條件變化）**：端側可執行模型的「能力密度」（capability-per-watt）在 2025-2026 跨越 Agent 任務門檻。 - **Q2（傳導機制）**：能力密度跨越門檻後，「執行位置」的決策權從「只能放雲端」變成「客戶可選擇」。客戶一旦有選擇權，就會基於成本（雲端 token 費）、延遲（Agent 多步往返）、隱私（企業資料、Meta token 排行榜外洩事件）三重壓力，將部分 workload 推向端側。 - **Q3（為何此時）**：三個臨界同時抵達——(a) 模型品質夠用、(b) 雲端推理成本已被量化為痛點（60 兆 token 事件）、(c) Agent 多步往返放大延遲問題（每步 200-500ms × 5-10 步 = 體驗崩潰）。任一缺一，趨勢無法成立。 - **Q4（反事實）**：若端側模型仍停在 2023 水準，雲端再貴客戶也只能忍受，趨勢不成立。→ 確認 Step 2 是真因。
✅ 宏觀（影響整個 AI 產業執行架構）✅ 長期（5 年以上）✅ 不可逆（模型壓縮技術不會倒退）
**產業趨勢**：AI 推理的執行位置正從「雲端集中」分裂為「雲-邊混合」，且 Agent 多步推理特性放大此分裂的經濟誘因。[推斷] #### Step 4：市場新機會浮現 - **Q1（競爭條件改變）**：原本「Agent 執行 = 雲端 GPU」的綁定被打破，端側出現「夠大、夠快、夠省電」執行 Agent 推理的需求。原有手機 AP（如 Snapdragon 8 Gen 3）的 NPU 是為「單次推理」設計（拍照、語音轉文字），不是為「Agent 多步推理 + 多模態並發 + 工具呼叫」設計。 - **Q2（傳導機制）**：Agent workload 對硬體的需求結構不同於傳統 AI——需要(a) 大 KV cache（記憶體頻寬）、(b) 多步低延遲（NPU 啟動成本必須低）、(c) 多模態並發（vision + speech + LLM 同時）。現有 NPU 在 (a)(c) 上設計不足。 - **為何現有競爭者未滿足**： - Qualcomm / MediaTek：架構鎖定，現有 NPU 是 2022-2023 規劃，KV cache / 記憶體子系統未為 Agent 優化。改架構需 18-24 個月。 - NVIDIA：商業模式不符，毛利模型建立在 H100/B100 級資料中心 GPU，端側 SoC（毛利 30-40%）對其稀釋。 - Apple：垂直整合但封閉，不對外供應。 - **Q4（反事實）**：若 Agent 不存在（只是單次推理），現有 NPU 夠用，機會不存在。→ 機會真實依附於 Agent 趨勢。 **市場新機會**：為「需在端側執行多步 Agent 推理 + 多模態並發」的裝置 OEM，提供針對 Agent workload 優化的 SoC（重點：記憶體子系統 + 低啟動成本 NPU + 多引擎並發排程）。 #### Step 5：目標客群與 JTBD **問題 A：為什麼是這個客群，而不是其他？** 候選客群： - 智慧眼鏡 / AR HMD OEM（如 Meta、Anduril 合作對象、LetinAR 客戶）✅ - 高階智慧手機 OEM ❌——已被 Qualcomm/MediaTek 強綁定，且現有 AP 仍可勉強應付，支付意願不夠強 - 智慧音箱 / 家用 Hub OEM ❌——使用情境延遲容忍高（雲端可接受），支付意願弱 - 車載資訊娛樂 ❌——車規認證週期太長（3-5 年），不是「此時」的機會 - 企業端 Edge Box（如零售、工廠）✅（次要客群） **因此目標客群是「智慧眼鏡 OEM」**，因為：(1) 眼鏡的延遲容忍最低（戴在臉上，超過 300ms 即不可用）、(2) 雲端往返完全不可行（隱私 + 連線不穩）、(3) 多模態並發是核心體驗（看 + 聽 + 語音回應同時）。 **問題 B：結構性條件使其有支付意願** 智慧眼鏡 OEM 面對「沒有 SoC 就沒有產品」的存亡壓力——Apple Vision Pro 用 M2（消費級耗電）證明現有手機 SoC 跑不動全天候 Agent 體驗。眼鏡 OEM 的差異化壓力 100% 來自「能否在 5-8W TDP 下執行 Agent」，這是「硬性需求」而非「優化需求」。 **問題 C：有能力的廠商為何未解決？** 選 **B（利益衝突）+ C（部分技術障礙）**： - Qualcomm 有 AR1 平台，但其架構仍以「單次推理 + 視覺處理」為核心，KV cache / 多模態並發排程器是 2024 年設計，要重新設計會傷害其手機 SoC 共用 IP 的攤提模型（利益衝突）。 - 改架構的具體障礙：記憶體子系統重設計需 LPDDR/HBM 介面與 NPU 的協同重做，約 18-24 個月。 **JTBD**：「在 6-8W TDP 預算內，於眼鏡裝置上執行 Agent 多步推理（5-10 步），同時保持多模態（vision + ASR + LLM）並發，且每步延遲 < 200ms，全天 8 小時使用。」 #### Step 6：痛點 - **具體環節阻礙**：Agent 多步推理中，每一步的「NPU wake-up + KV cache 載入 + 計算 + 寫回」循環。現有 NPU 為了省電採用「冷啟動」設計，wake-up 成本 50-100ms，5 步就是 250-500ms 純 overhead，未計入計算時間。 - **現有方案為何無法改善**：NPU 與記憶體子系統的耦合是 2-3 年前定型的，改善 wake-up 需要重設計 power gating 與 cache coherence 協議——這是矽層改動，不是韌體調整。 - **Q4（反事實）**：若 Agent 只需一步（如語音轉文字），現有方案夠用，眼鏡 OEM 不會找新方案。→ 痛點真實依附於 Agent 多步特性。 - **痛點深度**：眼鏡 OEM 對此痛點的支付意願極高——這是「能不能做出產品」的問題，不是「優化幾%」的問題。預估每 SoC 願付 $25-40（vs 手機 AP $15-25）。[推斷，需驗證] **核心痛點**：Agent 多步推理在現有 NPU 上的累積延遲與功耗，使眼鏡裝置無法達到「全天可用」的最低門檻。 #### Step 7：客戶價值 **解決方案**：專為 Agent workload 設計的 Edge AI SoC——含 (1) 持久化 NPU（無冷啟動）、(2) 大 KV cache 直連 LPDDR5X、(3) 多引擎並發排程器（vision NPU + LLM NPU + DSP 同時）。 - **機制**：透過 always-on NPU 設計消除 wake-up overhead；透過 KV cache 直連記憶體消除每步資料搬移；透過多引擎排程器讓多模態並發而非串行。 - **效益**：眼鏡 OEM 從「做不出可用產品」變成「可做出 8 小時續航的 Agent 眼鏡」——這是 0 vs 1，不是 +X%。 - **成本**：採用此 SoC 需重新設計 PCB、熱管理、軟體 stack。預估整合成本 $5-10M、12-15 個月。 - **效益 > 成本**：對眼鏡 OEM，這是「能否進場」的問題，整合成本相對於市場機會（Meta、Apple、各 OEM 在 2027-2028 都會推出眼鏡）可忽略。 - **競爭壁壘**：Qualcomm/MediaTek 需 18-24 個月改架構，時間窗口約 24-36 個月。 - **Q4（反事實）**：若痛點不存在，此方案只是「另一顆 NPU」，無客戶價值。→ 確認。 #### Step 8：商業價值 **商業模式**：賣 SoC + 配套 SDK（含 Agent runtime、模型量化工具、多模態排程 API），SDK 綁定客戶軟體 stack。 - **擷取機制**：SoC 單顆毛利 35-45%（高於手機 AP）＋ SDK 一旦整合，客戶軟體 stack 重寫成本 $10M+，形成轉換成本。 - **定價權來源**：(1) 架構先發優勢（早 18 個月）、(2) SDK 鎖定（客戶 Agent pipeline 適配後難遷移）、(3) 模型優化 know-how（特定壓縮配方）。 - **轉換成本組成**：客戶若改用對手 SoC，需重做 Agent runtime（6 個月工程）+ 重新量化模型（3 個月）+ 重新驗證多模態並發（3 個月）= 約 12 個月。 - **護城河動態**： - 加深機制：客戶用得越久，模型優化配方累積越多，遷移成本越高。 - 削弱威脅：Qualcomm 推出對等架構（預估 2027 H2-2028 H1）、開源 Agent runtime 標準化（降低 SDK 鎖定）。 - 頂峰：2027 中（先發優勢未被追上，且第一批客戶已深度整合）。 - **商業價值上限**：若窗口為 2026-2029（4 年），全球 AR/AI 眼鏡市場估計 2028 年 $10-15B [未驗證，建議搜尋 IDC/Counterpoint 2026.Q2 預測]，SoC BOM 佔比 15-20%，即 SoC TAM $1.5-3B/年。若拿下 20-30% 份額，年營收 $300-900M。 - **預估市場規模**：[未驗證，建議搜尋確認 2026 年 AI 眼鏡出貨預測] --- ### 機會 2：「RISC-V SSD / 儲存控制器 IP」——去 ARM 化的儲存子系統 #### Step 1：事實觀察 - [科技新報，2026.05]「三星 SSD 控制器導入 RISC-V，逐步降低對 ARM 依賴」→ 事實是：全球 SSD 控制器最大供應商之一正式採用 RISC-V。[來源] - [Bloomberg Tech，2026.05.18]「Kioxia 股價因 AI 驅動利潤激增而買盤湧入」→ 事實是：AI 推升 NAND/SSD 需求至「展望遠超預期」水準。[來源] - [Bloomberg Tech，2026.05.18]「芯片製造商 CXMT 銷售額飆升八倍」→ 事實是：中國記憶體廠營收 8 倍成長，IPO 在即。[來源] - [Tom's Hardware，2026.05.18]「創紀錄高價逼迫 SSD 和記憶體製造商借款 8.8 億美元」→ 事實是：儲存供應商現金壓力極大，需控制成本。[來源] #### Step 2：條件變化分析 - **改變的條件**：(1) ARM 授權費結構在 2024 年大幅調整（Arm Holdings 上市後對車用/儲存等垂直市場調整定價）、(2) RISC-V 在 2025-2026 達到「儲存控制器等級」的生態成熟度（成熟 toolchain、驗證 IP、量產案例）。 - **改變類型**：經濟性（ARM 授權成本上升）＋ 技術性（RISC-V 生態跨越成熟門檻）。 - **時間點**：兩個條件 2024 前都不成立——RISC-V 在儲存場景沒有量產驗證，ARM 授權還是業界默認選項。 - [推斷]：三星的決策需要 18 個月以上前置作業，意味條件變化在 2023-2024 即發生。 #### Step 3：趨勢導出 - **Q1**：CPU IP 的「壟斷溢價」可被取代的程度發生變化。 - **Q2**：當主要客戶（三星）證明 RISC-V 在量產可行，會在生態中產生「跟隨效應」——其他二線控制器廠（如 Phison、Silicon Motion）面對成本壓力會跟進，使 RISC-V IP / 服務市場需求暴增。 - **Q3**：為何此時——SSD 廠商現金壓力（$880M 借款）疊加 ARM 漲價，痛點被量化。 - **Q4**：若 ARM 沒漲價 + RISC-V 不成熟，三星不會動，趨勢不成立。 **產業趨勢**：嵌入式控制器（儲存、網通、IoT）的 CPU IP 採購正從「ARM 默認」進入「ARM/RISC-V 並列選項」階段。[推斷] #### Step 4：市場新機會浮現 - **Q1**：採購決策權從「沒得選」變成「可選擇」。 - **Q2**：客戶可選擇後，需要「RISC-V 控制器設計服務 + 驗證 IP + 軟體工具鏈」的供應商，這是現有 ARM 生態服務商（如 Arteris、Synopsys）沒有完整提供的空間。 - **為何未滿足**：ARM 生態服務商有商業模式衝突（他們的營收依賴 ARM 客戶），轉向 RISC-V 會稀釋自己。 - **Q4**：若 ARM 沒漲價，客戶不需要替代方案，機會不存在。 **市場新機會**：為儲存/網通控制器廠提供「ARM-to-RISC-V 遷移服務 + 驗證 IP 套件 + 客製化 RISC-V 核心 IP」。 #### Step 5：目標客群與 JTBD **問題 A**：為何是二線/三線 SSD 控制器廠（如 Phison、Silicon Motion、聯芸、得一），不是： - 一線廠（三星、Kioxia）❌——有內部團隊 - 網通設備廠 ❌——ARM 鎖定更深，遷移成本更高（次優先） - 中國控制器廠（如英韌、得瑞）✅（與台灣廠並列） **問題 B**：二三線控制器廠的支付意願來源——毛利壓縮（NAND 廠垂直整合擠壓）＋ 三星已驗證 RISC-V 可行（風險降低）+ 客戶（OEM）開始詢問 RISC-V 選項以降低 BOM。 **問題 C**：為何 ARM 生態服務商沒解決——商業模式衝突（B）。 **JTBD**：「在不犧牲性能與韌體相容性的前提下，將下一代 SSD 控制器從 ARM Cortex-R 系列遷移至 RISC-V，降低每顆 IP 授權成本 $0.5-2，且在 18 個月內完成首版投片。」 #### Step 6：痛點 - **環節阻礙**：RISC-V 在儲存控制器場景缺乏「成熟的驗證 IP + 韌體相容層 + 性能基準」，二線廠自行從頭做需 24+ 個月。 - **現有方案為何不解決**：ARM 不會做（自殺）；現有 RISC-V IP 廠（SiFive、Andes）以通用核心為主，未針對儲存特化（如低延遲中斷、ECC 加速協同）。Andes 較為接近但客製化服務能量有限。 - **痛點深度**：每顆 SSD 控制器若能省 $1-2 IP 費，年出貨百萬顆級別意味 $1-2M 直接利潤，痛點明確且可量化。 **核心痛點**：缺乏「儲存場景特化的 RISC-V IP + 遷移工具鏈」，使二線廠想跟進但無法在合理時間內完成。 #### Step 7：客戶價值 **解決方案**：儲存特化 RISC-V 核心 IP + ARM-to-RISC-V 韌體遷移工具 + 驗證套件。 - **效益**：客戶遷移時間從 24 個月縮短至 12 個月，IP 成本下降 $1-2/顆。 - **成本**：授權費 $0.5-1M + 5-10% royalty。 - **競爭壁壘**：先發者建立「儲存場景 RISC-V 設計知識」，後進者需 2-3 年累積。 #### Step 8：商業價值 - **擷取機制**：IP 授權費 + per-chip royalty + 設計服務。 - **定價權**：先發＋客製化深度。 - **護城河動態**：頂峰約 2027-2028，之後 RISC-V 生態普及後競爭加劇。 - **商業價值上限**：[未驗證，建議搜尋 2026 年 SSD 控制器出貨量] 全球 SSD 控制器年出貨估 5-7 億顆，若取得 10-15% 採用 RISC-V 路徑且收 $0.3 royalty/顆，年營收 $20-30M。市場有限但毛利極高。 --- ### ✅ 任務一因果鏈完整性審核

# 任務一因果鏈反事實審核 ## 審核表格
#箭頭反事實：拿掉 A 後 B 是否消失？消失的機制判定
1端側模型能力跨越門檻 → 雲-邊混合執行趨勢消失。若模型仍停在 2023 水準（70B+ 才可用），客戶無「執行位置」選擇權，只能放雲端客戶失去選擇權，雲端成本/延遲再痛也只能忍受✅
2雲端 token 成本壓力（Meta 60 兆事件）→ 客戶將 workload 推向端側**部分消失**。即使無成本壓力，延遲與隱私壓力仍可獨立推動端側化（眼鏡延遲不可妥協、企業資料隱私）成本只是三條件之一，非唯一驅動⚠️ 弱因果
3Agent 多步推理特性 → 現有 NPU 設計不足消失。若 Agent 只是單次推理（如語音轉文字），現有 NPU 的 KV cache、wake-up、多模態並發限制都不會成為瓶頸痛點直接依附於「多步」這個 workload 結構✅
4現有 NPU 為單次推理設計 → 出現 Agent 特化 SoC 機會消失。若現有 NPU 已能應付 Agent，OEM 無支付意願購買新 SoC機會的存在依賴「現有方案不夠用」✅
5眼鏡延遲容忍最低 + 雲端不可行 → 眼鏡 OEM 為最先付費客群消失。若眼鏡延遲容忍與手機相同，OEM 對 Agent SoC 的支付意願降至手機 OEM 等級（弱），客群浮現邏輯瓦解「戴在臉上 < 300ms」是物理性的硬約束，是支付意願的根源✅
6Qualcomm 共用 IP 攤提模型 → Qualcomm 不會優先重設計 NPU 架構**部分消失**。即使無攤提衝突，重設計仍需 18-24 個月技術時間。但「不優先」的決策動機確實來自利益衝突利益衝突解釋「為何不做」，技術障礙解釋「即使想做也來不及」——兩者皆需⚠️ 需拆解
7Agent SoC 先發 18 個月 → 24-36 個月競爭窗口**不消失（規模縮小）**。即使無先發，SDK 鎖定 + 模型優化 know-how 仍能提供部分護城河，但窗口會大幅縮短先發優勢是窗口的「乘數」，不是唯一來源⚠️ 弱因果
8ARM 授權費上升 + RISC-V 生態成熟 → CPU IP 多元化趨勢消失。若只有 ARM 漲價但 RISC-V 不成熟，客戶無替代選項；若只有 RISC-V 成熟但 ARM 不漲價，客戶無遷移誘因兩條件缺一則三星不會動✅
9三星採用 RISC-V → 二線控制器廠跟進壓力消失。若三星未驗證，二線廠面對「技術風險 + 成本壓力」會選擇繼續用 ARM（風險規避）三星案例提供「風險已被驗證」的信號，降低跟進方的決策成本✅
10ARM 生態服務商利益衝突 → 未提供 RISC-V 遷移服務消失。若無利益衝突（如 Synopsys 同時做 ARM 與 RISC-V 不衝突），他們會直接覆蓋此市場利益衝突是「為何空間存在」的核心解釋⚠️ 需驗證
11RISC-V 在儲存場景缺特化 IP → 二線廠自做需 24+ 個月消失。若已有特化 IP，二線廠採購即可，不需自做自做時間源於「沒有現成可買」✅
12儲存特化 know-how 累積 → IP 服務商的護城河**部分消失**。即使無 know-how 累積，先發者仍有客戶關係護城河，但會弱很多know-how 是護城河主要來源但非唯一⚠️ 弱因果
--- ## 修正 ### 修正 #2：雲端成本 → 端側化（弱因果） **修正**：「雲端 token 成本壓力推動 workload 推向端側」 **原因**：成本壓力是「三個並列驅動」之一（成本／延遲／隱私），不是充分條件。在原文 Step 3 的 Q2 中已將三者並列，但 Step 1 引用 Meta 60 兆 token 事件時隱含「成本是主驅動」的語氣，過度強調。 **正確陳述**：對眼鏡客群，延遲是主驅動（物理硬約束）；對企業客群，隱私是主驅動；成本壓力是放大器，不是觸發器。眼鏡 OEM 的支付意願**不依賴**雲端成本壓力。 **不受影響**： - 趨勢導出（Step 3）本身仍成立——三條件任一成立即足以驅動端側化 - 機會 1 的客群選擇（眼鏡 OEM）邏輯不變——其支付意願源於延遲而非成本 - 商業價值估算不變 --- ### 修正 #6：Qualcomm 利益衝突 → 不重設計（需拆解） **修正**：「Qualcomm 因利益衝突而不解決 Agent SoC 問題」 **原因**：原文將「為何 Qualcomm 沒解決」單一歸因於利益衝突（選項 B），但實際是 B + C 並存——利益衝突使其「不優先」，技術障礙使其「即使想做也需 18-24 個月」。 **正確陳述**： - B（利益衝突）解釋「為何 Qualcomm 不會在 2026 年立刻啟動全力重設計」 - C（技術時間）解釋「即使 Qualcomm 在 2026 啟動，最快也要 2027 H2 - 2028 H1 才有產品」 - 兩者疊加形成 24-36 個月窗口 **不受影響**： - 機會 1 的時間窗口估算（24-36 個月）不變，反而更穩固 - 護城河頂峰時間（2027 中）不變 - 目標客群與痛點分析不變 --- ### 修正 #7：先發 18 個月 → 24-36 個月窗口（弱因果） **修正**：「24-36 個月競爭窗口完全來自先發優勢」 **原因**：窗口的維持來源有三層——(a) 先發時間優勢、(b) SDK 與軟體鎖定、(c) 模型優化 know-how。即使先發優勢被追上，(b)(c) 仍可延長窗口。原文將窗口估算過度綁定於 (a)。 **正確陳述**：18 個月是「硬體先發」窗口；軟體與 know-how 鎖定可額外延長 12-18 個月。總窗口 24-36 個月由三層疊加而來，而非單一來源。 **不受影響**： - 商業價值估算（年營收 $300-900M）不變 - 護城河動態描述（加深 vs 削弱）不變 --- ### 修正 #10：ARM 生態服務商利益衝突（需驗證） **修正**：「ARM 生態服務商因利益衝突未進入 RISC-V 儲存特化服務」 **原因**：實際上 Synopsys、Cadence 等大型 EDA/IP 服務商**同時**提供 ARM 與 RISC-V 服務，不存在純粹利益衝突。原文歸因過強。 **正確陳述**：[未驗證] 真正原因可能是： - A. 儲存特化 RISC-V 市場規模（年 $20-30M）對 Synopsys 級廠商太小，不值得投入特化資源 - B. Andes 等中型 RISC-V IP 廠有意願但客製化能量受限於人力 需驗證資訊：(1) Synopsys/Cadence 目前在儲存 RISC-V 的具體服務深度、(2) Andes 在儲存場景的客製化案例數量。 **不受影響**： - 痛點本身（二線廠自做需 24+ 個月）仍成立 - JTBD 描述不變 - 三星採用 RISC-V 的觸發事件意義不變 - 商業價值估算的市場規模上限不變（但可能意味競爭比原本估計激烈） --- ### 修正 #12：know-how 累積 → 護城河（弱因果） **修正**：「儲存特化 know-how 是 RISC-V IP 服務商的主要護城河」 **原因**：護城河同時來自 (a) know-how 累積、(b) 客戶關係（IP 授權通常綁定多年）、(c) 驗證歷史降低客戶風險。原文偏重 (a)。 **正確陳述**：護城河是三層複合，know-how 是其中之一但非主導。 **不受影響**： - 護城河頂峰時間（2027-2028）不變 - 商業價值上限（$20-30M）不變 - 削弱威脅（RISC-V 生態普及）的判斷不變 --- ## 總結 - ✅ 7 個箭頭通過反事實測試（強因果） - ⚠️ 5 個箭頭為弱因果或需拆解，已提供修正 - ❌ 0 個箭頭完全不成立 **核心結論未動搖**：機會 1（Agent SoC for 眼鏡）與機會 2（RISC-V 儲存 IP）的商業邏輯仍成立。修正主要影響「歸因強度」與「窗口維持機制」，不影響「機會是否存在」與「目標客群是誰」。
