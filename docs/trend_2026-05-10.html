# SoC 策略分析報告 ## 分析期間：2026-05-03 ~ 2026-05-05 --- ## 任務一：結構性機會偵測 ### 機會 1：邊緣裝置端的 Agentic AI 推論 SoC——「離線可執行的代理運行時」 #### Step 1：事實觀察 - [Android Authority, 2026-05] ⭐「Google Gemma 4 在您的口袋中：如何完全離線執行最新 AI」→ 觀察事實：旗艦級模型已可在手機本地離線運行。 - [SemiEngineering, 2026-05] ⭐「邊緣 AI 無所不在、代理型、多模態且硬體加速」→ 觀察事實：產業論述已將「edge」「agentic」「multi-modal」「HW-accelerated」綁定為同一組關鍵字。 - [科技新報, 2026-05] ⭐「Meta 員工自建 AI 用量排行榜秀 30 天就破 60 兆 Token」→ 觀察事實：雲端 token 消耗已達內部都需要追蹤的量級，且資料外流風險導致排行榜「迅速撤下」——隱含雲端 inference 的隱私邊界正在被挑戰。 - [iThome, 2026-05-04]「Google 推 Gemini Enterprise Agent Platform，Vertex AI 服務整併至新企業 AI 代理平臺」→ 觀察事實：雲端大廠把 agent 平台定位為「整合層」，但 agent 的執行端點未必都在雲。 - [The New Stack, 2026-05] ⭐「透過 Claude Managed Agents，Anthropic 希望為您運營 AI Agent」→ 觀察事實：Anthropic 把 agent 從「模型」轉為「託管服務」，意味著 agent 是「持續運行的進程」而非「一次性 prompt」。 #### Step 2：條件變化分析 - **條件變化**：2024 年中以前，手機端只能跑「壓縮版 helper 模型」（如語音轉文字）；2025 下半到 2026Q1，旗艦模型（Gemma 4、Apple Intelligence、Gemini Nano）的離線可執行門檻已落在 8–12B 參數區間，且 NPU 算力（>40 TOPS）+ LPDDR5X 頻寬已達到可承載 agentic loop（多步推理）的最低水位。 - **變化性質**：技術性（NPU/記憶體頻寬）+ 經濟性（雲端 token 成本對企業已成可見負擔，60 兆 token/30 天的內部消耗即為證據）+ 監管性（Meta 排行榜資料外流事件、企業合規）。 - **時間點**：2024 年中以前，agent loop 在手機本地跑會耗盡電量且延遲不可接受；2026Q1 起，特定 use case 的「本地 agent 一次完整 loop」已落在使用者可忍受範圍內。 - [推斷]——個別事實有來源，但「門檻已跨越」是綜合推論。 #### Step 3：趨勢導出 - Q2 機制：NPU 算力門檻跨越 → 模型可在本地完成「多步推理」 → 改變了「agent 必須住在雲端」的產業預設架構 → 雲廠商與晶片廠的價值分配重構。 - Q3 此時門檻：兩年前（2024）即使有 NPU，模型品質不足以完成 agentic loop（會出現幻覺中斷）；2026 起 Gemma 4 等模型「在 8B 級別已可完成多步任務」——這是品質的臨界點，不是算力的臨界點。 - Q4 反事實：如果 Gemma 4 / Apple Intelligence 級別的小模型品質沒有突破，即使 NPU 算力翻倍，agent 在本地也只能做單步推理，趨勢不成立。所以驅動因子是「小模型品質×NPU 算力」的乘積跨越門檻，缺一不可。 - **趨勢陳述**：AI agent 的執行位置正在從「雲端集中」分裂為「雲端編排 + 邊緣執行」的雙層架構。
- 宏觀性 ✅（影響整個運算產業）長期性 ✅（5 年以上）不可逆性 ⚠️（雲端 inference 成本若大幅下降可能逆轉，標記 [推斷]）
#### Step 4：市場新機會浮現 - Q1 競爭條件變化：原本 SoC 競爭條件是「TOPS 數字 + ISP + modem」；現在新增一條「能否在本地穩定運行 agent runtime（多步 inference + tool calling + 記憶體常駐 context）」。 - Q2 機制：agent 不同於單次 inference——它需要長時間常駐記憶體（KV cache）、頻繁的小型 inference 而非一次大型 inference、且需要與 OS/app 的 tool calling 介面。現有 SoC 為「峰值 TOPS 優化」，不是為「持續低延遲小批次 inference 優化」。 - 現有競爭者為何未滿足： - Qualcomm/MediaTek 的 NPU 架構為「大批次峰值算力」設計，KV cache 管理仰賴 DRAM，agent 持續運行時記憶體頻寬被吃滿。 - 雲廠商（Google/Anthropic）有架構鎖定——他們的商業模式建構在 token 計費上，主動把 inference 推到邊緣會傷害自己 ARR（利益衝突）。 - Q4 反事實：如果趨勢不存在（agent 仍住雲端），這個機會不存在。 - **機會陳述**：為「邊緣 agent runtime」優化的 SoC IP / 子系統——特別是 KV cache 專用 SRAM、低延遲小批次 NPU 排程器、與 OS-level tool calling 介面的硬體加速。 #### Step 5：目標客群與 JTBD **問題 A：為什麼是這個客群？** 候選客群： - ❌ 旗艦手機 OEM（Samsung、Xiaomi）：他們會用 Qualcomm/MediaTek 的整合 SoC，不會單獨採購 agent runtime IP。 - ❌ 雲廠商自研晶片（Google Tensor、Apple Silicon）：他們是垂直整合，不向外採購。 - ✅ **企業級 AI PC / 行業終端 ODM**（如工業手持機、車載資訊娛樂、企業協作裝置 OEM）：他們有合規壓力（不能把客戶資料送雲），但沒有自研 SoC 能力。 **問題 B：結構性條件** - 合規壓力（Meta 60 兆 token 事件 + 各國資料在地化）讓「資料不出裝置」成為硬需求，不只是 nice-to-have。 - 差異化壓力：企業終端 OEM 在硬體規格戰已輸給消費品牌，必須在「企業合規 AI 能力」上找新定位。 **問題 C：有能力者為何沒解決？** - Qualcomm：選擇不做（B）——他們的 AI Hub 策略仍偏雲端模型 porting，把 edge agent 當作消費功能，不是合規賣點。 - MediaTek：技術上能做但商業模式不符（C+B）——他們追求大量出貨的消費 SoC，不會為小批量企業終端做專用 IP。 - NVIDIA：技術上能做但業務重心在雲端 H100/B200，邊緣是 Jetson 線——而 Jetson 剛宣布部分產品 EOL（見 Tom's Hardware 報導），顯示其在邊緣的優先級在下降。 **JTBD**：「在無法連雲或不可連雲的情境下（醫院、工廠、金融、跨境出差），讓終端使用者完成原本需要雲端 agent 完成的多步任務（文件草擬、跨應用整合、決策輔助），並且通過企業 IT 的資料合規稽核。」 #### Step 6：痛點 - 具體環節阻礙：現有 SoC 跑 agent 時，KV cache 在 DRAM 與 NPU 之間反覆搬運，造成（1）延遲不一致（從 50ms 跳到 500ms）（2）電池快速耗盡（3）多 app 共用 NPU 時的 context switching 開銷。 - 現有方案為何無法改善：[推斷] NPU 架構在 RTL 層級為「大 tile、大批次」優化，KV cache 是後加上去的「軟體解法」，需要重新設計 dataflow 才能根治。 - Q4 反事實：如果延遲與耗電問題解決，企業 OEM 就不需要新方案——所以痛點真實。 - 痛點深度：[推斷] 企業終端 OEM 對「能通過 IT 部門合規驗證的 on-device agent」有顯著支付意願，因為這直接影響其招標資格（醫療 HIPAA、金融 PCI、政府 FedRAMP）。 - **核心痛點**：現有 SoC 無法在「持續 agent loop」場景下同時滿足延遲、功耗、多任務隔離三條約束。 #### Step 7：客戶價值 **解決方案**：提供「Agent Runtime 子系統 IP」——包含 KV cache 專用 SRAM tile、persistent context manager、低延遲小批次 NPU scheduler、tool calling 硬體介面——以 IP 或 chiplet 形式授權給企業終端 ODM/OEM。 - 機制：KV cache 移入 on-die SRAM 消除 DRAM 搬運 → 延遲與功耗同步下降；硬體 tool calling 介面消除 app 切換開銷。 - 效益：[推斷] agent loop 延遲 P99 從 500ms 降到 100ms 級，電池續航從 2 小時 agent 使用提升到 6 小時以上。 - 成本：IP 授權 + 整合工程約 12–18 個月、單顆 BOM 增加 [未驗證]；學習成本中等。 - 效益 > 成本機制：對企業 OEM 而言，這是「招標資格」級別的差異化，不是性價比競爭——支付意願不對標消費 SoC 而是對標合規認證費用。 - 競爭壁壘：[推斷] Qualcomm/MediaTek 重設計 RTL 約需 18–24 個月（一個 SoC 世代），時間窗口存在。 - Q4 反事實：痛點不存在則價值消失——成立。 #### Step 8：商業價值 **商業模式**：IP 授權 + per-chip royalty + 共同設計服務費。 - 擷取機制：IP 一次性授權建立關係 → royalty 隨客戶出貨綁定 → 軟體 SDK 鎖定客戶開發 ecosystem。 - 定價權來源：（1）整合進客戶 SoC 後，轉換成本包含 18 個月重新設計 + 軟體 SDK 重寫；（2）合規認證綁定，客戶換 IP 等於重新認證。 - 護城河動態： - 加深機制：SDK 累積 agent recipe（每個垂直行業的最佳實踐）→ 後續客戶更難切換。 - 削弱威脅：（a）Qualcomm/MediaTek 18–24 個月後出原生方案（b）開源 agent runtime（如 ONNX Runtime + agent extension）拉平 IP 優勢。 - 護城河頂峰：[推斷] 2027 下半年到 2028 上半年，約 18–24 個月窗口。 - 商業價值上限：[未驗證，建議搜尋確認] 企業終端 SoC 市場全球年出貨估約 1.5–2 億顆，若取 10% 滲透 + $2 royalty，年營收約 $30–40M 級別，加上前期 IP 授權費總體 windowed value [未驗證]。 - 預估市場規模：[未驗證，建議搜尋確認 2026 企業終端 SoC TAM] --- ### 機會 2：RISC-V 控制器在「主控之外」的特用 SoC 滲透——以儲存控制器為突破口 #### Step 1：事實觀察 - ⭐ [科技新報, 2026-05]「三星 SSD 控制器導入 RISC-V，逐步降低對 ARM 依賴」→ 觀察事實：大廠首次在量產儲存控制器產品線換掉 ARM IP。 - [Tom's Hardware, 2026-05-04]「Nvidia 在亞洲供應鏈的曝險比例達到生產成本的 90%」→ 觀察事實：地緣政治對 IP 來源敏感度上升。 - [Tom's Hardware, 2026-05-04]「嚴重短缺推動 SSD 與 HDD 長期供應協議達五年紀錄」→ 觀察事實：儲存控制器是長期確定性需求。 - [iThome, 2026-05-04]「美國戰爭部與 8 大科技巨頭簽署 AI 協議」→ 觀察事實：軍規/政府市場對 IP 主權的需求上升。 #### Step 2：條件變化分析 - 條件變化：（1）RISC-V 工具鏈成熟度在 2025–2026 跨過「生產級可用」門檻；（2）ARM 授權費結構改動（IPO 後 royalty 上升、訂閱模式）讓大客戶尋找替代；（3）地緣政治讓「IP 主權」從口號變成 BOM 決策因素。 - 時間點：2023 年 ARM 的授權爭議（Qualcomm vs ARM）和 2024 年 IPO 後的策略變化是觸發點。 - [來源] 三星行動為直接證據；其他為 [推斷]。 #### Step 3：趨勢導出 - Q2 機制：ARM 商業條款變動 → 大客戶評估替代 → RISC-V 工具鏈剛好成熟 → 從「邊角應用（IoT 玩具）」開始滲透到「成本敏感大量產品線（控制器）」→ 隨成功案例累積，向上滲透。 - Q3 此時門檻：工具鏈成熟（GCC/LLVM、Linux 支援、debug 工具）+ 商業誘因（授權費）同時存在，2022 年只有商業誘因但工具鏈不行，2024 年工具鏈剛成熟但案例不足，2026 三星案例提供「可複製性證明」。 - Q4 反事實：如果 ARM 沒改授權條款，即使 RISC-V 工具鏈成熟，遷移成本仍高於收益——所以商業條款是觸發因子，技術成熟是必要條件。 - **趨勢陳述**：RISC-V 從「外圍 MCU」向「特用 SoC 控制核心」滲透。
- 宏觀性 ✅長期性 ✅不可逆性 ✅（一旦客戶完成遷移，反向成本更高）
#### Step 4：市場新機會浮現 - Q1 競爭條件變化：原本 SoC 內的控制核心預設 ARM Cortex-R/M；現在「RISC-V 控制核心 + 完整軟體生態」成為可投標選項。 - Q2 機制：三星案例證明「儲存控制器這類量大、軟體相對封閉、不需 app ecosystem」的場景，RISC-V 可替代——同類場景（網卡、SSD、感測器 hub、車用控制）將跟進。 - 現有競爭者為何未充分滿足：ARM 不會自我蠶食；商業 RISC-V IP 廠（SiFive、Andes）規模小，缺乏「特用 SoC 完整子系統」打包能力——機會在於「特定垂直的 RISC-V SoC 子系統供應商」而非通用 IP。 - Q4 反事實：趨勢不在則無此機會——成立。 - **機會陳述**：為儲存/網路/感測器 hub 等「軟體封閉型」特用 SoC 提供「RISC-V 控制核心 + 周邊 + RTOS」打包解決方案的供應商。 #### Step 5：目標客群與 JTBD **問題 A：為何是這個客群？** 候選： - ❌ 旗艦 AP SoC 廠（QCT、MTK）：app ecosystem 鎖死，不會換。 - ❌ MCU 廠（ST、NXP）：他們自己會做 RISC-V，不採購。 - ✅ **儲存/網路控制器 ASIC 廠 + 二線車用 MCU/特用 SoC 廠**：軟體封閉、量大、ARM 授權費敏感。 **問題 B：結構性條件** - 控制器 ASIC 毛利率被擠壓（NAND 廠不斷壓價）+ ARM 授權費上升 → BOM 結構必須優化。 - 中國客戶有 IP 主權需求，台廠若能提供 RISC-V 方案可拿到原本拿不到的訂單。 **問題 C：有能力者為何沒解決？** - SiFive 規模太小（A）+ 商業模式不符（C）：他們賣 core IP，不打包整個子系統。 - 大 ASIC 廠（Marvell、Phison）：選擇不做（B）——三星案例是 in-house，不是採購；他們會自己做不會買。 - **真正機會**：給「沒能力 in-house 做 RISC-V 但有控制器產品線」的二線廠（中國、台灣特用 SoC ODM）。 **JTBD**：「在 ARM 授權費上升 + IP 主權需求 + NAND/網卡毛利壓力下，二線控制器 ASIC 廠需要在 12 個月內導出可量產的 RISC-V 控制核心子系統，且軟體相容性風險可控。」 #### Step 6：痛點 - 阻礙：自行整合 RISC-V core + 周邊 + RTOS + debug 工具鏈需要 24+ 工程師年；二線廠沒有這個資源池。 - 現有方案為何無法改善：SiFive 賣 core 不賣子系統；Andes 工具鏈但軟體 stack 需自組；開源方案驗證風險高。 - Q4：如果有 turnkey RISC-V 控制核心子系統，客戶就不必自組——痛點真實。 - **核心痛點**：缺乏「turnkey、可量產驗證、有客戶背書」的 RISC-V 控制核心打包方案。 #### Step 7 & 8：（與機會 1 結構類似，精簡） **客戶價值**：縮短 RISC-V 導入時程 24 月 → 12 月、消除驗證風險（reference design 已過量產）、保留 ARM 退路（dual core 設計）。 **商業價值**：IP 授權 + 共同設計服務 + 長期 maintenance contract。護城河來自「跑過量產驗證」的 reference design 累積——每多一個客戶量產，下個客戶風險折扣越大。 - 護城河頂峰：[推斷] 2028 年前，之後大廠 in-house 能力補齊。 - 商業價值上限：[未驗證] 全球儲存/網路/特用控制器 SoC 年出貨數十億顆，但能用「外部 turnkey」的可定址市場 [未驗證，需搜尋確認]。 --- ### ✅ 任務一因果鏈完整性審核

--- ## 修正 ### 修正 1：Meta 事件 → 合規壓力（因果反向） 修正：「Meta 60 兆 token 資料外流事件 → 企業合規壓力上升至招標門檻級別」 原因：因果方向反了。GDPR (2018)、HIPAA、PCI-DSS、各國資料在地化法規早於 Meta 事件存在，是這些**既有監管**讓企業 IT 把 Meta 事件解讀為「警訊」。Meta 事件是「症狀證據」，不是「成因」。把 Meta 事件當成驅動因，會誤導對時間軸的判斷（誤以為 2026Q1 才是合規壓力起點，實際上已存在 5+ 年）。 正確陳述：「既有資料保護監管（GDPR/HIPAA/各國在地化）長期存在 + 雲端 AI 採用率在 2025-2026 跨過企業 IT 治理門檻 → 合規部門開始把 AI 列入既有合規框架 → Meta 事件成為內部說服管理層的『典型案例』」 對機會 1 的影響：合規驅動力的**時間起點要前移**，意味著： - 此機會的「時間窗口」可能更短（合規需求早已存在，只是缺技術方案——一旦方案出現，採用速度可能比預期快） - Meta 事件不能作為「趨勢剛開始」的證據，應視為「趨勢已成熟到出現公開事件」的證據 不受影響： - 「合規壓力是企業 OEM 支付意願的根本來源」此因果仍成立 - 「邊緣 agent runtime 的技術門檻 2026 才跨越」此判斷不變 - 機會 1 的目標客群識別不變 ### 修正 2：雲端 token 成本 → 雙層架構不可逆（因果過度簡化） 修正：「雲端 token 成本壓力 → 雙層架構不可逆」 原因：不可逆性的來源是多重獨立動因（隱私、合規、離線可用性、延遲、token 成本），任一存在即足以維持邊緣執行需求。將「不可逆性」綁定在「token 成本」單一動因上，會導致一個錯誤的監測策略——若雲端成本下降就以為趨勢逆轉，忽略其他動因仍在作用。 正確陳述：「雙層架構由四個獨立動因共同支撐（隱私、合規、離線、延遲），其中合規與離線在可預見未來不會消失 → 即使 token 成本大幅下降，趨勢仍不可逆，但**邊緣執行的滲透率上限會被壓低**（成本敏感場景回流雲端）」 對機會 1 的影響： - 趨勢的「方向」不可逆，但「規模」會受雲端成本影響 - 商業價值上限的估算需要分場景：合規驅動部分（高支付意願、規模小）vs 成本驅動部分（規模大、價格敏感） - 監測指標應分別追蹤：（a）合規法規變化（b）雲端 inference 單價趨勢 不受影響： - 機會 1 的目標客群（合規驅動的企業終端 OEM）不受影響——他們的支付意願不來自 token 成本 - 技術門檻判斷不變 - 護城河分析不變 ### 修正 3：QCT/MTK 重設計時間 → 時間窗口存在（因果不完整） 修正：「QCT/MTK 重設計 RTL 需 18-24 月 → 時間窗口存在」 原因：窗口的存在不只取決於「他們做這件事需多久」，更取決於「他們是否選擇做」。若 QCT/MTK 因利益衝突或商業優先級選擇不做，窗口可能比 24 月更長；若他們判斷此為策略要地全力投入，窗口可能比 18 月更短。原因果只考慮技術時間，忽略商業決策。 正確陳述：「時間窗口 = min(技術重設計時間, 商業決策時間)。技術上 18-24 月為下限；商業上他們是否優先投入取決於『企業終端市場規模對其營收的重要性』——目前 QCT/MTK 主力為消費市場，企業終端 < 10% 營收占比 [未驗證]，因此商業決策時間可能更長（24-36 月）。」 對機會 1 的影響： - 時間窗口估算應分情境：保守 18 月（QCT 視為戰略要地）、基準 24 月、樂觀 36 月 - 監測指標：QCT/MTK 的企業終端營收占比變化、其在 MWC/Computex 是否將 enterprise edge 列為主軸 不受影響： - 護城河存在的判斷不變 - 機會 1 的目標客群與痛點不變 - 商業模式設計不變 ### 修正 4：二線廠評估 RISC-V → 需要 turnkey（因果不完整） 修正：「二線廠評估 RISC-V → 需要 turnkey 方案」 原因：二線廠對 turnkey IP 的需求是**結構性的、長期存在的**，不是 RISC-V 創造的。把因果寫成「RISC-V → turnkey 需求」，會誤判市場規模：實際情況是二線廠對 ARM 子系統的 turnkey 需求也一直存在（ARM 設計服務商如 Open-Silicon 即在服務此需求）。 正確陳述：「二線廠**長期**缺乏 IP 整合能力 → 一直依賴 turnkey 方案 → 過去用 ARM
