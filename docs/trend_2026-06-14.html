# SOC 策略分析報告 ## 分析期間：2026.06.03 - 2026.06.09 --- # 任務一：結構性機會偵測 ## 機會 1：邊緣裝置上的「離線可執行 LLM」推論 SoC ### Step 1：事實觀察 - [Android Authority, 2026.06]「Google Gemma 4 在您的口袋中：如何完全離線執行最新 AI」[來源] → 觀察事實：Google 釋出可在手機端離線執行的小型 LLM Gemma 4 - [SemiEngineering, 2026.06]「邊緣 AI 無所不在、代理型、多模態且硬體加速」[來源] → 觀察事實：產業論述已從「邊緣 AI 是否可行」轉為「邊緣 AI 必須硬體加速」 - [科技新報, 2026.06]「Meta 員工自建 AI 用量排行榜秀 30 天就破 60 兆 Token，但資料外流迅速撤下」[來源] → 觀察事實：企業內部 AI 用量爆炸，且「資料外流」是即時觸發的撤下原因 - [iThome, 2026.06.08]「OpenAI 為 ChatGPT 引進 Lockdown 模式，透過限制部分工具存取降低提示注入攻擊風險」[來源] → 觀察事實：雲端 LLM 服務商開始用「限制功能」來處理資安問題 - [Tom's Hardware, 2026.06.08]「美國新建 AI 數據中心三分之二建在缺水地區」[來源] → 觀察事實：雲端 AI 基礎設施已遇到實體資源（水、電）瓶頸 ### Step 2：條件變化分析 **改變的條件**：「在裝置端執行有用的 LLM」從「技術不可行」轉為「技術可行 + 經濟有必要」 - **時間點**：2025 下半年 ~ 2026 上半年 - **改變類型**：技術 + 經濟雙重變化 - 技術面：3B-8B 參數模型透過量化（INT4/INT8）已可在 ~8GB RAM 行動裝置上以可接受延遲執行（Gemma 4、Phi 系列等） - 經濟面：雲端 Token 成本（Meta 案例：30 天 60 兆 Token）+ 資料外流風險（Meta 撤下事件）+ 雲端基礎設施實體瓶頸（缺水缺電）三者疊加 - **多久前不存在**：2023 年端側模型品質不足以實際使用；2024 年成本壓力與資料外流案例尚未達臨界規模 - [推斷] 標記：兩個面向皆有來源支持，但「臨界點達成」為推斷 ### Step 3：趨勢導出 **Q2 機制**：條件變化透過「企業 IT/裝置 OEM 的採購決策邏輯改變」傳導 - 過去邏輯：所有 AI 推論 → 雲端（因為端側做不到） - 新邏輯：敏感資料 + 高頻使用的推論 → 端側；複雜推理 → 雲端（混合架構） - 傳導路徑：採購邏輯改變 → 對 SoC 規格要求改變 → 整個產業的 NPU 規格從「能跑 1B 模型」轉為「能跑 7B 模型且支援 KV-cache 管理」 **Q3 為何此時足夠**：三個臨界點同時跨越——(a) 模型品質達實用門檻（Gemma 4、Apple Intelligence 已實際出貨）、(b) 資料外流成為董事會級議題（Meta 案例）、(c) 雲端成本對企業可見化（Token 計費透明）。任一單獨不夠，三者疊加才形成不可逆。 **Q4 反事實**：若 Gemma 4 級模型不存在，企業仍會因成本與資料外流尋找替代方案，但只能選擇「自建私有雲」（成本更高），趨勢將延後 2-3 年但不會消失。**因此 Gemma 4 是加速器而非唯一原因**，趨勢具不可逆性。 **因此，產業趨勢是**：終端裝置必須具備「離線執行 3B-8B 模型」的能力，雲-端混合架構成為新預設 驗證：✅ 宏觀（影響整個行動 + PC + 嵌入式產業）✅ 長期（5 年以上）✅ 不可逆（資料主權監管疊加會強化此趨勢） ### Step 4：市場新機會浮現 **Q1 競爭條件改變**：SoC 的「AI 算力規格」從「附加功能」變為「採購決策的第一層篩選條件」 - 過去：CPU 性能 → GPU → ISP → NPU（NPU 是 nice-to-have） - 現在：NPU TOPS + 記憶體頻寬 + 模型支援度 → 其他（NPU 是 must-have，且需可驗證） **Q2 傳導機制**：OEM 需要在規格表上向消費者/企業客戶展示「可離線跑哪些模型」，這逼迫 SoC 廠商必須提供：(a) 足夠 NPU 算力（>40 TOPS）、(b) 高記憶體頻寬（LPDDR5X 或以上）、(c) 完整 model zoo 支援與工具鏈 **現有競爭者為何沒充分滿足**： - Qualcomm/MediaTek 旗艦 SoC 算力足夠，但**集中在高階手機（>$600）** - **中階市場（$200-400 手機 + 中階筆電 + IPC/Edge Box）存在空白**：這些裝置量大、單價低、毛利薄，旗艦 SoC 用不起，但又需要 LLM 能力 - 結構性原因：旗艦 SoC 廠商的成本結構與 IP 授權模式不適合做「低成本但高 NPU 占比」的產品 **Q4 反事實**：若無「裝置端 LLM」趨勢，中階 SoC 只需漸進升級 NPU，不存在新空間。趨勢消失則機會消失。 **因此，市場新機會是**：「中階消費電子裝置（$200-500 BOM）的 AI-first SoC」——專為 3B-8B LLM 推論優化、犧牲部分傳統 GPU/ISP 性能換取 NPU 與記憶體頻寬 ### Step 5：目標客群與 JTBD **問題 A：為什麼是「中階 Android OEM + 企業 Edge Device OEM」，而不是其他客群？** 排除： - ❌ **旗艦手機 OEM（Samsung、小米旗艦線）**：已綁定 Qualcomm/MediaTek 旗艦，轉換成本高且不缺算力 - ❌ **Apple 生態**：垂直整合，不採購第三方 SoC - ❌ **超低階手機（<$150）**：BOM 容不下任何 NPU 升級，使用者也無支付意願 - ❌ **資料中心 GPU 客戶**：屬於雲端推論市場，已被 NVIDIA 鎖定 **因此目標客群是**：(a) 中階 Android 手機 OEM（Transsion、realme、中階小米/OPPO 線）、(b) 企業 Edge AI 設備 OEM（工業 IPC、零售終端、會議室設備） **問題 B：結構性支付意願來源** - (a) 中階手機 OEM：**差異化壓力**——中階市場高度同質化，「離線 AI 助理」是少數能形成規格差異的賣點 [推斷] - (b) 企業 Edge OEM：**合規壓力**——歐盟 AI Act、各國資料主權法規要求敏感資料不出境，雲端 LLM 直接違規 [推斷，建議查證具體條款] **問題 C：有能力的廠商為何沒解決？** 選擇 **「商業模式不符」**： - Qualcomm Snapdragon 8 Gen 系列毛利結構建立在「旗艦定位 + 授權費」上，做中階低毛利 SoC 會稀釋平均 ASP 與股東預期 - MediaTek Dimensity 9000 系列同理 - 機制：若大廠下殺中階 AI SoC，會傷害自己旗艦 ASP（內部蠶食），這是真實的利益衝突 - 因此他們會做「中階版」但會刻意保留 NPU/記憶體頻寬差距以保護旗艦定位 → 留下空間 **因此，JTBD 是**：「中階裝置 OEM 在 12-18 個月內，需要推出能在規格表上明確標示『支援離線 Gemma 4 / Llama 3.1 8B』的產品，達到雲端方案 70% 的使用者體驗，但 BOM 成本控制在旗艦的 40% 以下」 ### Step 6：痛點 **JTBD 執行中的具體阻礙**： - 現有中階 SoC（如 Snapdragon 7 Gen 系列、Dimensity 7000 系列）的 NPU 算力（~15-25 TOPS）+ 記憶體頻寬不足以流暢執行 7B 模型，**首 token 延遲 >3 秒、生成速度 <5 token/s**，使用者體驗破裂 **現有方案為何無法改善**： - 中階 SoC 的記憶體子系統是為「中度遊戲 + 多工」設計，記憶體頻寬通常 <50 GB/s - LLM 推論是**記憶體頻寬瓶頸**而非算力瓶頸（每個 token 需讀完整模型權重一次） - 機制：在現有架構下加 TOPS 沒用，必須重新設計記憶體子系統，但這會牽動整個 die area 配置 → 不是「未優化」而是「架構決策已鎖定」 **Q4 反事實**：若現有中階 SoC 能跑 7B 模型流暢，OEM 不會找新供應商。痛點真實存在。 **痛點深度**：[推斷] 中階 OEM 願意為「可標示離線 LLM 支援」的 SoC 多付 20-30% 溢價，因為這是少數能拉開競品差距的賣點 **因此，核心痛點是**：中階 SoC 的記憶體頻寬架構無法支撐 7B 級 LLM 即時推論，但 OEM 又無法承擔旗艦 SoC 的成本 ### Step 7：客戶價值 **解決方案**：專為 LLM 推論優化的中階 SoC——犧牲傳統 GPU 性能 30-40%，換取 LPDDR5X 高頻寬接口 + 大型 NPU + 模型專用 cache hierarchy - **機制**：透過 weight streaming 與 KV-cache on-chip SRAM 設計，使 7B INT4 模型在 50 GB/s 頻寬下也能達 >15 token/s - **效益**：[推斷] OEM 可標示「離線 Gemma 4 / Llama 3.1 8B 支援」→ 規格差異化 → ASP 提升 $30-50 - **成本**：BOM 增加 $8-12（額外 SRAM + 高速記憶體接口）+ 軟體生態建置成本 - **效益 > 成本機制**：3-5 倍槓桿（增 $10 BOM 換 $30-50 ASP） - **競爭壁壘**：Qualcomm/MediaTek 因利益衝突在 18-24 個月內不會推出真正競品（會出但會保留差距）。窗口 ~24 個月 - **Q4 反事實**：若痛點不存在，OEM 不需要付溢價，方案無價值。痛點真實，價值成立 ### Step 8：商業價值 **商業模式**：晶片銷售 + 模型工具鏈授權（類似 NVIDIA CUDA 模式但聚焦邊緣） - **擷取機制**：晶片毛利（30-40%）+ 工具鏈年費 + OEM 軟體支援收入 - **定價權來源**： - 工具鏈鎖定：OEM 一旦整合特定 SoC 的 LLM runtime，轉換成本包含模型重新量化、應用層 API 重寫、QA 流程重建——[推斷] 約 6-9 工程師月 - 模型生態：與 Hugging Face / Google Gemma 團隊建立的 model zoo 適配關係 - **護城河動態**： - **加深機制**：每多一個 OEM 採用 → 模型生態貢獻者增加 → 後來者轉換成本上升 - **削弱威脅**：Qualcomm/MediaTek 一旦決定犧牲旗艦 ASP 來下殺中階，2-3 年內可追上 - **頂峰時間**：2027 年中至 2028 年初（在大廠決策反應之前） - **商業價值上限**：[未驗證，建議搜尋確認] 中階 Android SoC 年出貨 ~5 億顆，假設拿到 5% 市占 + ASP $25 = 年營收 $625M - **預估市場規模**：[時間不明，建議搜尋確認] 全球中階智慧型手機 SoC 市場 2026 約 $15-20B [未驗證] --- ## 機會 2：「Agent-Safe」MCU/MPU——專為 AI Agent 執行設計的安全邊緣處理器 ### Step 1：事實觀察 - [The New Stack, 2026.06]「Anthropic 推出 Claude Managed Agents」+「Claude Code 一次運行 5 個智能體」[來源] → AI Agent 從概念進入大量併發執行階段 - [The New Stack, 2026.06]「『一個危險的組合』：能夠『破壞』AI agent 工作流的 2 個因素」「傳統 IAM 模式需要重新評估以適應 AI agent 驅動的工作流」[來源] → 業界已意識 Agent 的 IAM 模型是缺失的 - [iThome, 2026.06.08]「OpenAI 為 ChatGPT 引進 Lockdown 模式」「Anthropic 揭露惡意帳號濫用 Claude 趨勢，中度風險以上 AI 輔助攻擊者占比已逾五成」[來源] → Agent 安全已成現實威脅 - [Bloomberg, 2026.06]「Bessent 稱 Anthropic 的 Mythos 是中美 AI 競賽中的突破性進展」[來源] → Agent 能力進入下一代 - [iThome, 2026.06.08]「多倫多大學研究示警，駭客可用免費 AI 模型提高蠕蟲程式擴散能力」[來源] → Agent 武器化進入主流關注 ### Step 2：條件變化分析 **改變的條件**：AI Agent 從「實驗性 demo」變為「實際在企業環境中執行高權限任務」 - **時間點**：2026 Q1-Q2 - **改變類型**：使用模式變化 + 安全威脅實體化 - **多久前不存在**：2025 年 Agent 仍以「展示能力」為主，真正大規模執行（修改檔案、呼叫 API、跨系統操作）是 2026 才開始 - [來源] + [推斷] 混合 ### Step 3：趨勢導出 **Q2 機制**：Agent 執行的特殊性（自主決策 + 高權限 + 不可預測行為）使得「在通用 OS 上執行 Agent」變成本質性的安全問題。傳統的 sandbox 與 IAM 模型假設「執行者是人類或確定性程式」，無法處理「執行者是隨機性 LLM」的情境。 **Q3 為何此時足夠**：(a) Claude/ChatGPT 等已實際讓 Agent 動手操作真實系統、(b) 攻擊者已開始系統性濫用（Anthropic 報告：中度以上風險帳號 56%）、(c) 企業合規部門開始拒絕無隔離的 Agent 部署 [推斷] **Q4 反事實**：若 Agent 仍是 demo，趨勢不存在。但 2026 已過實體化臨界。 **因此，產業趨勢是**：「Agent 執行環境需要硬體級隔離與行為驗證」成為企業部署的剛性需求 驗證：✅ 宏觀 ✅ 長期 ⚠️ 不可逆性：依賴於「Agent 實用化」持續，若 Agent 泡沫破裂則趨勢減弱 → [推斷] ### Step 4：市場新機會浮現 **Q1 競爭條件**：邊緣設備（企業伺服器、工業閘道、智慧型設備）的處理器規格新增「Agent 隔離域」要求 **Q2 機制**：企業 IT 採購開始要求「Agent 工作負載必須在獨立安全域執行」→ 推動硬體級 TEE（Trusted Execution Environment）+ 行為監控加速器 **現有競爭者為何沒充分滿足**： - ARM TrustZone 是為傳統機密計算設計，不是為「LLM 隨機行為」設計 - Intel SGX/TDX 是資料中心級，不適合邊緣 - 結構性原因：現有 TEE 假設「程式行為可預測」，但 Agent 不可預測，需要新的執行時行為監控架構 **Q4 反事實**：若無 Agent 趨勢，TEE 漸進演化即可。 **因此，市場新機會是**：「邊緣 Agent 執行專用 SoC/MCU」——內建 LLM 行為監控、權限沙箱、強制策略執行的處理器 ### Step 5：目標客群與 JTBD **問題 A**： - ❌ **超大型雲端服務商**：自建客製晶片 - ❌ **消費電子**：使用者不在乎企業級隔離 - ❌ **傳統 IPC 客戶**：尚未部署 Agent **因此目標客群是**：(a) 企業 Edge AI 設備 OEM（會議室 AI、零售 Agent、工業自動化）、(b) 受監管產業（金融、醫療）的內部 Agent 平台 ODM **問題 B 支付意願**：合規壓力——金融/醫療/政府部署 AI Agent 必須通過稽核，硬體級隔離是「可勾選的稽核項目」[推斷，建議查證具體法規進度] **問題 C 既有廠商為何沒解決**： 選擇 **「技術障礙 + 商業模式不符」**： - 技術障礙：LLM 行為監控需要硬體加速 + 即時推論（監控者本身也是模型），這是新的計算範式 - 商業模式：ARM/Intel 的授權與市場模式不適合做「合規導向小量高毛利」產品 **因此，JTBD 是**：「企業 Agent 部署方在 18 個月內，需要可向稽核員證明『Agent 行為在硬體層級被監控且越權立即阻斷』的執行平台」 ### Step 6：痛點 **阻礙**：現有 Agent 執行平台無法在硬體層產生「不可篡改的行為審計軌跡」，純軟體方案在稽核時被認為不足 **現有方案為何無法改善**：軟體層審計可被有 root 權限的攻擊者繞過；現有 TEE 無 Agent 語義理解能力（不知道「Agent 試圖讀檔」是否合法） **Q4 反事實**：若軟體方案能滿足稽核，市場不需新硬體。但 [推斷] 受監管產業稽核員已開始質疑純軟體方案 **因此，核心痛點是**：缺乏硬體強制的、LLM-aware 的執行隔離與審計機制 ### Step 7：客戶價值 **解決方案**：內建 Agent Behavior Monitor 的 SoC——硬體級監控所有 LLM 對外呼叫、權限請求、資料存取，並可在硬體層強制阻斷 - **機制**：dedicated security island + 即時策略引擎 + 不可篡改 log buffer - **效益**：通過稽核（不通過 = 不能部署 = 損失整個專案） - **成本**：BOM 增 $5-15 + 軟體整合成本 - **競爭壁壘**：[推斷] 12-24 個月，直到 ARM 推出 TrustZone Agent Extension 或 Intel 推出 TDX-Agent - **Q4 反事實**：痛點消失則無價值；痛點受監管驅動，短期內不會消失 ### Step 8：商業價值 - **擷取機制**：高毛利 SoC（受監管市場可承受 50%+ 毛利）+ 安全策略平台訂閱 - **定價權**：認證壁壘（一旦通過 FedRAMP / HIPAA 等認證，OEM 不會輕易換） - **護城河**：認證資產隨時間累積 → 加深；但若 ARM/Intel 推出標準化方案則被取代風險高 - **頂峰時間**：2027-2029 - **市場規模**：[未驗證，建議搜尋確認] 企業 Edge AI 設備市場 2026-2030 CAGR [需查證] --- ### ✅ 任務一因果鏈完整性審核

# 任務一因果鏈反事實測試審核 ## 機會 1：中階 AI-first SoC
箭頭反事實：拿掉A後B是否消失？消失的機制判定
Gemma 4 等可用小模型 → 端側 LLM 趨勢**不消失**，因為企業仍會因成本與資料外流尋找替代（自建私有雲），趨勢延後但不消失驅動力是「成本+資料主權」三因子，模型可用性只是其一⚠️ 部分成立——應修正為「加速因素」非「必要因素」
雲端成本 + 資料外流 + 雲端實體瓶頸（三因子）→ 端側 LLM 趨勢消失，因為若雲端便宜、安全、無限擴展，OEM 無動機推端側OEM 採購邏輯由成本與合規驅動✅
端側 LLM 趨勢 → NPU 從附加變必須消失，若無端側需求，NPU 仍為 nice-to-haveOEM 規格表優先級重排✅
NPU 變必須 → 中階市場空白出現**不完全消失**，中階市場 NPU 升級壓力本就存在（影像處理、語音），LLM 只是放大中階市場差異化壓力是長期存在的⚠️ 應修正為「LLM 放大既有空白」而非「LLM 創造空白」
旗艦 SoC 廠商利益衝突 → 不下殺中階消失，若無內部蠶食風險，Qualcomm/MediaTek 會直接做商業模式鎖定✅
中階 SoC 記憶體頻寬架構鎖定 → 7B 模型跑不順消失，若記憶體子系統可重設計，痛點消失物理頻寬瓶頸✅
「可標示離線 LLM 支援」→ ASP 提升 $30-50**未驗證消失與否**——這是 [推斷]，缺乏中階手機規格差異化定價的實證假設 OEM 能將規格轉為 ASP，但中階市場價格戰激烈，可能無法擷取❌ 因果不成立，需修正
工具鏈整合 → 6-9 工程師月轉換成本**未驗證**——此數字為推斷，無來源工具鏈鎖定強度未經驗證❌ 因果強度不明
OEM 採用 → 模型生態網路效應加深護城河**不確定消失**，模型生態主要由 Hugging Face / Google 等中立方主導，未必綁定特定 SoC邊緣 LLM 生態的網路效應強度未經驗證❌ 因果不成立，需修正
## 機會 2：Agent-Safe MCU/MPU
箭頭反事實：拿掉A後B是否消失？消失的機制判定
Agent 實用化 + 攻擊實體化 → 硬體隔離需求消失，若 Agent 仍為 demo 或無攻擊，企業無採購動機合規與威脅雙驅動✅

受監管產業合規壓力 → 願付硬體溢價**部分未驗證**——「稽核員拒絕純軟體方案」為 [推斷]，缺乏具體法規或稽核案例假設稽核標準會演進到要求硬體隔離❌ 因果強度不明，需補證據
ARM/Intel 商業模式不符 → 不做合規導向小量產品**不完全消失**——ARM 已有 Confidential Compute Architecture，可能延伸至 Agent 場景大廠可能透過 IP 擴展而非新產品線進入❌ 因果不成立，需修正
認證資產累積 → 護城河加深消失，若無認證制度，無壁壘但認證制度本身為 [推斷] 仍未成形⚠️ 依賴前提未驗證
--- ## 修正 ### 修正 1：機會 1 的 ASP 提升假設 **修正**：「中階 SoC 可標示離線 LLM 支援 → OEM ASP 提升 $30-50」 **原因**：中階手機市場為價格敏感型，規格差異化能否轉為 ASP 溢價未經驗證。Transsion、realme 等品牌的歷史顯示，新規格通常被用於「同價位多賣」而非「漲價」。 **修正後陳述**：[推斷] 中階 SoC 的「離線 LLM 能力」更可能轉為**市占率擴張**（同價位搶單）而非 ASP 提升。SoC 廠商的商業價值來自**出貨量增長**而非單顆溢價。 **不受影響**： - 端側 LLM 趨勢本身（由三因子驅動） - 中階 SoC 記憶體頻寬痛點存在 - 旗艦廠商的利益衝突分析 - 任務一機會 2 的所有推導 ### 修正 2：機會 1 的工具鏈鎖定強度 **修正**：「工具鏈整合 → 6-9 工程師月轉換成本 → 強護城河」 **原因**：6-9 工程師月為無來源推斷。實際上，若 SoC 廠採用標準化的 ONNX / GGUF / MLIR 等開放格式，轉換成本可能僅 1-2 工程師月。 **修正後陳述**：[未驗證] 工具鏈鎖定強度取決於「SoC 廠選擇專有 vs 開放格式」的策略。若採專有格式，鎖定強；若採開放格式，鎖定弱但生態大。需搜尋驗證主流邊緣 LLM runtime（如 MLC、llama.cpp、ExecutorTorch）的可移植性實況。 **不受影響**： - 端側 LLM 趨勢、中階市場痛點分析 - 「時間窗口」概念本身（仍存在，但寬度待驗證） ### 修正 3：機會 1 的模型生態網路效應 **修正**：「OEM 採用 → 模型生態貢獻者增加 → 後來者轉換成本上升（網路效應）」 **原因**：邊緣 LLM 的「模型生態」主要由 Hugging Face、Google、Meta 等中立模型發布方主導，並非由 SoC 廠主導。模型作者沒有動機特別優化某顆 SoC，網路效應弱。 **修正後陳述**：護城河來源應重新定位為**「OEM 整合經驗 + 軟體 SDK 成熟度」**而非「模型生態網路效應」。這是線性而非指數型護城河，加深速度較慢。 **不受影響**： - 時間窗口的存在（但護城河頂峰可能更早出現，2027 中而非 2028） - 客戶價值分析 ### 修正 4：機會 1 中「LLM 創造中階市場空白」 **修正**：「端側 LLM 需求 → 中階市場空白出現」 **原因**：中階 SoC 的差異化壓力與 NPU 升級需求本就存在（影像處理、語音），LLM 是「放大器」而非「創造者」。 **修正後陳述**：端側 LLM **將原本緩慢的中階 NPU 升級壓力，加速為迫切的架構重構需求**。機會本質從「創造新市場」修正為「加速現有市場結構性轉變」。 **不受影響**： - 機會仍真實存在，但「機會的新穎性」應下調 - 目標客群、JTBD、痛點分析不變 ### 修正 5：機會 2 的合規壓力強度 **修正**：「受監管產業合規壓力 → 願為硬體隔離付溢價」 **原因**：「稽核員拒絕純軟體方案」目前為純推斷。歐盟 AI Act、NIST AI RMF 等框架雖提及隔離，但未明確要求硬體級。 **修正後陳述**：[未驗證，建議搜尋確認] 此機會的時間軸高度依賴「稽核標準從原則性要求演進為技術性要求」的速度。建議搜尋： - 2025-2026 NIST AI RMF 是否新增硬體隔離條款 - 金融業（FFIEC）對 Agent 部署的指導意見 - 已部署 Agent 的企業實際遇到的稽核要求 **不受影響**： - Agent 實用化趨勢本身（多源證據） - 現有 TEE 架構不匹配的技術分析 - 機會 1 全部分析 ### 修正 6：機會 2 的 ARM/Intel 不進入假設 **修正**：「ARM/Intel 商業模式不符 → 不做 Agent-Safe 處理器」 **原因**：ARM 已有 Confidential Compute Architecture (CCA)，可透過 IP 擴展進入此市場，未必需新產品線。Intel TDX 同理。「商業模式不符」的論證不足。 **修正後陳述**：[推斷] 真正的時間窗口可能來自**「大廠 IP 擴展的開發週期」**（通常 18-30 個月）而非「他們不會做」。機會本質為**時間窗口套利**而非結構性壟斷。 **不受影響**： - Agent 隔離需求的存在 - 技術障礙分析 - 但商業價值上限應下調，因為護城河時間更短 --- ## 整體影響摘要
